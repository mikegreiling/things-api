## Inventory: locale reliance (UIC1 evidence)

Nearly ALL leaf actions are **title-pinned English** today: every menu item (Items menu, Repeat submenu, context menus, presumably the `…` menu) shares the generic `performCommand:` identifier; Repeat-dialog controls carry only volatile `_NS:NNN` numbers (the interval field's id even changes with the frequency). Locale-INDEPENDENT: the confirm-sheet buttons (`action-button-1` — the only semantic AXIdentifiers found), structural addressing (roles, `window 1`/`sheet 1`, index-in-parent), and the menu-bar top-level ids (`items`/`file`/`edit` — but not their leaves). Doctrine: **pin the app to English on the deployment machine** (`defaults write com.culturedcode.ThingsMac AppleLanguages -array en`) + the recipe canary's proven fail-closed refusal under locale drift (German negative test, UIC1-c). Optional follow-up: an explicit preflight locale check in the driver.

## The plan — three probe campaigns, then builds

### UIC2 (VM campaign) — ✅ DONE 2026-07-15 ([lab/uic2-project-menu.md](../lab/uic2-project-menu.md))
(a) **Project `…`-menu discovery + drive** — ✅ **feasible, but HYBRID not pure-AX.** The `…` **button is not an AX node** (drawn inside the title text-area frame; the Items menu bar has no Repeat for a shown project; `AXShowMenu` on the title gives a text-edit menu). The `…` menu opens only by a **synthetic mouse click**; once open its items are AX-**readable** (by `description`: Complete Project/When/Add Tags/Add Deadline/Move/**Repeat…**/Duplicate/Delete/Share) with resolvable `AXPosition` **but INERT to `AXPress`** — actuation needs a synthetic click. The Repeat **dialog** it opens is a **sheet, byte-identical to the to-do dialog** (freq `_NS:116`, OK `_NS:164`) and **fully AX-drivable, backgrounded, no focus steal**. All four project ops DB-verified: **make** (new template `fu`=256 + instance, identity replacement), **reschedule** (repeat-bar ▸ Change…, `fu` 256→8, identity preserved), **pause/resume** (repeat-bar ▸ Pause↔Resume). Repeating-project reschedule/pause/resume/**stop** live in an **always-visible repeat bar** in the project view — **no card double-click** (unlike to-dos). `open -g` reveals+selects **without activating** (frontmost stayed Finder). `…`/popover navigation is **foreground-bound** (synthetic mouse); the dialog sheet is backgroundable.
(b) **Two-step create-repeating-project** — ✅ validated (`things:///add-project` → promote via (a) → template + spawned instance).
(c) **CRASH1** — ✅ **REPRODUCED, deterministic 2/2.** Stop-repeat a project → select the demoted project → `EXC_BREAKPOINT`/SIGTRAP (Swift trap, `ThingsModel`). **Transient** (gone after relaunch), **no data loss**; crash is on the SELECT not the Stop. Two `.ips` banked; oddities §7 **C5**. → any `project.stop-repeat` op must NOT reveal the demoted result.
(d) **`todo.stop-repeat` re-check** — ✅ **no new surface** (Items ▸ Repeat still = Reschedule/Pause/Show Latest, no Stop; the project repeat bar has no to-do analog — to-do Stop stays card-double-click-only). **Recommend DROP** (ruling #3).
**Net for the builds:** project repeat ops are buildable but depend on a **mouse-synthesis primitive (NATIVE1)** to open+navigate the custom `…` menu / repeat-bar popover (target the AX-resolved item positions, never `AXPress`); the dialog half rides the existing AX machinery. `project.make-repeating` additionally needs the `…` **button** position, which has no AX node (title-length-dependent) — the reschedule/pause/resume/stop ops all use the AX-resolvable repeat bar instead.

### AXDRAG1 (VM campaign) — ✅ DONE 2026-07-15 ([lab/axdrag1-reorder.md](../lab/axdrag1-reorder.md))
Hardened the NATIVE1 area-drag into an arbitrary-rank recipe and scoped tag reorder. **Verdicts:** **(a) precision placement WORKS** both directions incl. to-first/to-last (DB-verified via `TMArea."index"`). Drop geometry — sidebar is 40px/slot (h24 row + h16 spacer); **upward drags use static target coords, downward drags subtract one slot-height** because lifting the source collapses its slot (lower rows shift up 40px). Index rewrite: first drag renumbers the WHOLE list (sparse/irregular); each later drag reassigns ONE index (intermediate, not exact midpoint) and **may renumber a neighbour rather than the dragged row** → undo captures the full order. **(b)** scroll-wheel synthesis (`CGEventCreateScrollWheelEvent`→HID tap) reveals off-viewport rows; the sidebar **AXScrollBar `AXValue` (0–1) is readable** so scroll distance is computable. **(c) mid-drag auto-scroll works bidirectionally** but is time-based → prefer pre-scroll for a deterministic landing (full list exceeds one viewport). **(d)** areas can't nest (no structural hazard on a mis-drop — only wrong rank); **Escape mid-drag aborts cleanly** (the op's abort primitive). **(e) TAG reorder FEASIBLE** (Window▸Tags AX menu press; `AXDialog`; `TMTag."index"`+`parent`): **drop-on-centre re-parents, drop-on-boundary reorders** — a mis-aimed drop silently NESTS, and tag names are not AX-exposed (address by position). **(f) reversible** via pre-rank capture + drag-back (order restores; index values don't). Note the divergence from NATIVE1: AppleScript-seeded sidebar rows carry an EMPTY `AXDescription` — identity is via descendant `AXStaticText`. This was a PROBE campaign; the production `area.reorder-sidebar`/`tag.reorder` ops are not built.

### NATIVE1 (spike; gates AXDRAG1's drag path and todo.stop) — ✅ DONE 2026-07-15 ([lab/native1-spike.md](../lab/native1-spike.md))
The row-selection/mouse-synthesis primitive. ~~Try in order: (1) **JXA ObjC bridge** ... (2) a small compiled Swift helper.~~ **VERDICT: Tier 1 (JXA ObjC bridge) wins — no Swift helper.** `osascript -l JavaScript` resolves AX frames (`AXUIElementCopyAttributeValue` for `AXPosition`/`AXSize` via `ObjC.castRefToObject` + `CFCopyDescription`) and synthesizes mouse via `CGEventCreateMouseEvent`. **Double-click → the card opens** (repeat bar reachable → `todo.stop-repeat` unblocked). **Drag → an area reorders** (writes `TMArea."index"`, 0 → −448). BUT ~~`CGEventPostToPid`~~ is inert for Things — the working post is `CGEventPost(kCGHIDEventTap)`, which is **foreground+unlocked only** (backgrounded: NO; under-lock: NO — opposite of the AX menu path). **TCC: the Accessibility grant alone authorizes CGEvent posting** (no second consent class; no PostEvent/ListenEvent row created). `AXScrollToVisible` UNAVAILABLE on rows (−25206) → AXDRAG1 scroll edge case needs synthesized scroll-wheel events.
