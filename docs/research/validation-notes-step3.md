## Ancillary Findings

| date (YYYY-MM-DD) | topic | finding | impact |
|---|---|---|---|
| 2026-03-12 | Inbox exit semantics | Inbox items can have tags, deadlines, and checklists without leaving Inbox. Assigning an area, assigning a project, or assigning a `when` value removes the item from Inbox and reclassifies it into normal task lists (`Anytime`, `Today`, `Upcoming`, etc.) based on schedule state. | Important for capture/clarify workflow design and for interpreting side effects of update commands. |
| 2026-03-12 | URL visibility in UI | Successful URL mutations may not open a new Things window, may not unminimize an existing window, and may not navigate any visible window to the affected item/list. | Visual confirmation is unreliable; read-back verification is mandatory. |
| 2026-03-12 | Unsupported command modal behavior | Unsupported-command and missing-auth modals stack until dismissed manually. Valid subsequent URL commands can still execute while a modal is open. | Modal presence is a UI hazard, not a global mutation lock. |
| 2026-03-12 | Record links and IDs | Both to-dos and projects are addressed via `things:///show?id=<UUID>`. Promoting a to-do to a project assigns a new ID; the old to-do ID no longer resolves. The resulting error text says `Cannot show the list with ID ... because it does not exist.` | Promotion should be treated as identity replacement, not in-place type mutation. Any cross-links or audit keys must be updated accordingly. |
| 2026-03-12 | Case sensitivity | Tag names and area names are matched case-insensitively in tested URL updates. | Normalize names in tooling, but do not rely on case to distinguish taxonomy entries. |
| 2026-03-12 | Project completion semantics | URL-based project completion does not mirror the manual UI prompt. In the tested no-heading case, it implicitly marked unresolved child to-dos as completed, preserved already-canceled children, and completed the project with no prompt. | High-risk mutation. Agent tooling must inspect child state first and require explicit policy before issuing project completion. |
| 2026-03-12 | Logbook timing | User-observed but not yet formally validated: completed items may be moved into Logbook immediately in some cases or later by app maintenance/overnight processing unless manually logged. | Add follow-up validation if automation must mutate already-completed or already-logged items. |
| 2026-03-12 | Resolved project reopening | Any action that causes an unresolved child to exist inside a previously completed, canceled, or already logged/archived project reopens that project. This includes adding a new open child and manually un-completing an existing child. Re-resolving all children does not automatically re-complete the project; the project must be explicitly completed/canceled again. | Project state is not purely declarative or automatically restored. Automation must reissue explicit project resolution after child-state changes when that is desired. |
| 2026-03-12 | Checklist independence from to-do state | Completed or canceled to-dos can contain unresolved checklist items. Adding checklist items to a completed to-do does not reopen it, and changing to-do completion/cancelation status does not automatically alter checklist-item state. | Checklist items are lightweight substeps, not status-governing child tasks. Automation should not infer parent task state from checklist state. |
| 2026-03-12 | Heading parameter semantics | In tested URL mutations, `heading=<name>` on `add` did not create a missing heading; it simply created the to-do without a new heading. | Heading creation and heading placement are distinct concerns. Tooling should assume headings must already exist before using heading-placement parameters. |
| 2026-03-12 | Heading targeting by name | `heading=<name>` works as a placement parameter on `update` when the target heading already exists inside the current/target project. Matching is case-insensitive. If `list-id` is omitted, heading lookup only works within the to-do's current project; it does not implicitly move an external to-do into another project just because a heading with that name exists there. Duplicate heading names would be ambiguous because headings do not expose shareable IDs in the UI. | Heading placement is usable by name, but heading identity is weak. Avoid duplicate heading names inside a project if automation needs deterministic placement. |
| 2026-03-12 | Area links | Areas expose `things:///show?id=<UUID>` links via the UI, not just name-based references. | Area mutations can use either name or ID, which is useful for deterministic automation. |
| 2026-03-12 | Repeating-item mutation hazard | Repeating to-dos accept some non-scheduling mutations (for example `title`, checklist content), but attempting to set `when` reproducibly crashes Things and leaves the item unchanged. | Repeating items require field-level write guards. Do not treat them like normal to-dos. |
| 2026-03-12 | Auth token retrieval via `things.py` | `things.token()` returned the current Things auth token and matched the user-provided token. | Auth token discovery appears automatable via local reads, which reduces setup friction but bypasses the intended manual-secret flow. Treat as optional medium-friction/internal behavior, not guaranteed public API. |
| 2026-03-13 | Trash representation | Read-only inspection of existing trashed items showed that Trash is represented as task state (`trashed=1` in `TMTask`), not as an obvious normal destination list. Trashed tasks retained their original `project` / `area` associations in SQLite, and `things.py` exposes them via `trash()` / `tasks(..., trashed=True)` rather than as a regular list assignment. | Current evidence suggests Trash is a special state transition, not something that can be targeted through ordinary `list` / `list-id` URL parameters. If we need stronger proof, run a controlled before/after on one disposable item manually moved to Trash. |
| 2026-03-13 | SSH with active GUI session | User confirmed that `open "things:///..."` works from an SSH session as long as the same user is already logged into the macOS GUI on that machine. Logged-out behavior remains unvalidated. | Remote invocation is viable in a logged-in desktop session, but "fully headless/logged-out" remains unsupported until proven otherwise. |
| 2026-03-13 | Project view composition and "later items" | The project UI is not just the active project to-dos. It also includes future-dated items grouped by date, repeating items with their next occurrence, Someday items at the end of the later-items section, a toggle labeled `Hide later items`, and a separate logged-items section with its own hide toggle. `things.py` can expose the scheduled (`upcoming`), Someday (`someday`), logged (`logbook`), and trashed (`trash`) project-associated items, but the repeating project item `hello-world` was only visible in raw SQLite, not in `things.py` project/list helpers. | Any agent attempting to mirror the native project view needs a composite read model, not a single `project.items` call. `Later items` should be treated as a real UI concept spanning scheduled, repeating, and Someday project children. |
| 2026-03-13 | Heading structural model | In the live heading test, heading-contained to-dos carried `heading` / `heading_title` and, in raw SQLite, no longer had a direct `project` value; they were related to the project through the heading. `things.py` compensates for this in `tasks(project=...)`, so headed child to-dos are still returned when filtering by project. The heading itself is a separate `heading` record with its own ordered `items`. | Headings are true sub-containers, not decorative labels. A heading-agnostic agent can still operate on child to-dos by UUID/project query, but should preserve heading metadata so grouping is not lost. |
| 2026-03-13 | `projects(include_items=True)` duplication with headings | When a project contains headings, `things.projects(uuid=..., include_items=True)` returned heading-contained to-dos twice: once as flat entries in `project.items`, and again nested under the heading record's `items`. | Do not use `project.items` as the canonical flat child-task list when headings exist. Prefer `tasks(type='to-do', project=<PROJECT_ID>, ...)` for a flat deduplicated child-task feed, and `tasks(type='heading', project=<PROJECT_ID>, include_items=True, ...)` only when grouping is explicitly needed. |
| 2026-03-13 | Heading UI actions | User-observed UI actions on headings include: promote heading to project, move heading (with its child to-dos) to another project, archive heading, and drag-drop reorder of the heading together with its children. Archiving presents a prompt analogous to project completion when unresolved child to-dos exist. | Headings are operational containers in the UI, not mere visual separators. Even if automation ignores heading edits, existing heading structure can materially affect how users perceive project organization and order. |
| 2026-03-13 | `This Evening` representation | After moving `[AI-GTD-TEST] ModalProbe` to `This Evening`, raw SQLite showed `start=1` (Today/Anytime-style), `startDate=<today>`, and `startBucket=1`. The item remained visible in `things.today()` and `things.get(...)`, but `things.py` did not expose any explicit `evening` flag; it looked like a normal Today item with a very large positive `today_index`. Querying live data showed only a small set of active items with `startBucket=1`, consistent with the `This Evening` subsection. | A custom API that needs to distinguish `Today` vs `This Evening` should read `startBucket` directly from SQLite (or equivalent raw DB field), because `things.py` does not surface this distinction cleanly. |
| 2026-03-13 | Reorder primitive: `Today` -> `This Evening` | In a controlled six-item probe, updating `today task 1` from `Today` to `This Evening` via `when=evening` moved it to the top of the `This Evening` section, ahead of the prior `evening task 1/2/3` order. | Cross-section moves appear to prepend into the target section rather than append. This is a promising primitive for synthetic reordering, but needs more probes before being treated as reliable. |
| 2026-03-13 | Reorder primitive: `This Evening` -> `Today` | In the same controlled probe, updating `evening task 3` from `This Evening` to `Today` via `when=today` moved it to the top of the `Today` section, ahead of the prior `today task 2/3` order. | Cross-section moves prepend into `Today` as well. | 2026-03-13 | Same-section reorder via bounce | A two-step bounce of `today task 2` (`Today -> This Evening -> Today`) moved it to the top of the `Today` section while leaving the relative order of the other tested Today items unchanged. | This establishes a viable synthetic reorder primitive for Today/Evening sections: bounce an item through the opposite section to front-insert it into the original section. | 2026-03-13 | Sequence-level Today reorder | A second bounce of `today task 3` (`Today -> This Evening -> Today`) produced the expected constructed order in `Today`: `today task 3`, `today task 2`, `evening task 3`. | Reordering within `Today` is algorithmically viable: repeated bounce operations can build a deliberate target order by front-inserting items in reverse desired order. | 2026-03-13 | Sequence-level Evening reorder | A bounce of `evening task 2` (`This Evening -> Today -> This Evening`) moved it to the top of the `This Evening` section, yielding the expected Evening order: `evening task 2`, `today task 1`, `evening task 1`. | Reordering within `This Evening` is also algorithmically viable via the same front-insert bounce primitive. |
| 2026-03-13 | Today/Evening order representation in DB | The six-item probe now validates directly from SQLite without needing UI inspection. `startBucket=0` items formed the `Today` subsection and sorted by ascending `todayIndex` as `today task 3`, `today task 2`, `evening task 3`. `startBucket=1` items formed the `This Evening` subsection and sorted by ascending `todayIndex` as `evening task 2`, `today task 1`, `evening task 1`. The items' `index` values did not change during these Today/Evening moves. | For Today/Evening contexts, the custom API can validate order directly from DB using `(startBucket, todayIndex)` while treating `index` as a separate structural ordering field. | 2026-03-13 | Sparse ordering keys | Observed `todayIndex` values were not dense counters like `0, 1, 2`. They were sparse integers with large gaps (for example `-1731`, `-1074`, `-650` for `Today` and `6000626`, `6000887`, `6001652` for `This Evening`). Across all bounce-based reorder probes, only the moved item's `todayIndex` / `startBucket` changed; neighboring items kept their prior values. The general `index` field also uses sparse integer keys rather than dense sequence numbers. | Things appears to use sortable rank keys rather than renumbering whole lists on each move. Bounce-based reorder writes therefore seem localized to the moved item, though occasional internal compaction/renormalization remains possible in theory and has not been observed yet. |
| 2026-03-13 | Collision handling remains unresolved | Active collisions do exist for ordering keys, but current samples are insufficient to claim a universal tiebreaker. For `Anytime`, same-`todayIndex` peers appeared ordered consistently with `index`, which supports the view that `todayIndex` is probably not relevant outside Today-style contexts. For same-`index` peers, the available examples were confounded by repeating/later-item behavior or too small a sample to treat as definitive. | The custom API should use a comparator per view/context and keep tie-breaking rules provisional until controlled same-context collision tests are run on disposable non-repeating items. Do not encode a global fallback like title or UUID yet. |
