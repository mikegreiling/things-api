# UI-vector research — the dedicated-Mac GUI write vector (UI1 → UI2)

The "ui" vector is the FOURTH write vector (after URL scheme / AppleScript / Shortcuts), flagged **most-disruptive tier**: a real Things GUI driven by VNC synthetic input (hardware-level HID — no TCC, no Accessibility, which is SIP-blocked in the golden). **Mike greenlit exploring it 2026-07-14** as the hybrid create-quiet / transform-loud doctrine: create and edit content through the quiet vectors, and reach for GUI driving ONLY for the operations that have no other spelling.

- **UI1** (feasibility + the deadline-less-repeat discriminator) lives in [s-campaign-results.md](s-campaign-results.md) "UI-vector campaign (UI1)". Headline: `File → New Repeating To-Do` is VNC-drivable end-to-end; a real `rt1_recurrenceRule` template row lands.
- **UI2** (this doc): the operations UI1 identified as UI-only — making an existing item repeat, editing/pausing/stopping a rule, and to-do/heading → project conversion — plus the disruption profile Mike asked for, plus two folded-in DB micro-probes (UI2-g, UI2-h).

Run: [`lab/scripts/research-ui2.sh`](../../lab/scripts/research-ui2.sh). ONE `--vnc-experimental` clone `things-run-ui2-20260714-013427` (airgapped, clock-pinned 2026-07-05, Things 3.22.11 / macOS 15.7.7 / DB v26), autonomous, driven adaptively (each `vncdo capture` read to choose the next click). Discovery campaign — no assertions; ground truth is the DB row deltas, corroborated by the screenshot sequence (`00`–`32e`) + `final.sqlite` under the run's artifacts dir (gitignored). Reuses the SX6/UI1 VNC mechanics: `vncdo` via a throwaway venv; **menus close between separate vncdo sessions** (chain a menu-nav inside ONE invocation); popup dropdowns are picked by reading the open menu and clicking the item; **`vncdo` ⌘-chords do not register** (see UI2-e).

**`fu` (frequency unit) codec observed:** daily = 16 (UI1), **weekly = 256**, **monthly = 8** (+ an `of` offset array). tp=0 = fixed, ts=0 = same-day, rrv=4, fa=1 across all.

**Incidental app-quirk confirmations:** the URL `update` endpoint is to-do-only — pointing it at a **project** id (`things:///update?id=<project>&completed=true`) raises a "Things URL Scheme … does not exist" error modal and no-ops (projects need `update-project`); these URL-scheme error modals are Things-owned windows that steal focus (the known json/url error-modal oddity). Completing a project via **AppleScript** cascade-completes its children including heading rows.

## UI2-e — the disruption profile (the question Mike asked)

VNC synthetic input is delivered at the **hardware HID level to absolute screen coordinates** — it targets whatever is on screen, not a specific app. Four empirical facts bound what that means for driving Things:

1. **The menu bar belongs to the frontmost app.** Activating Finder (`osascript … Finder activate`) swapped the menu bar to Finder's menus (`27e-finder-frontmost.png`); Things' **Items → Repeat / Convert** menus are simply not present unless Things is frontmost. Since the reliable driving method is menu clicks (see #3), **menu-driving requires Things frontmost**.
2. **A click on a visible Things control raises Things (click-through = focus steal).** Clicking a Things window control while Finder was frontmost brought Things to the foreground (`lsappinfo front` → Things). So a coordinate click either lands on the other app (menu bar / its windows) or yanks Things forward.
3. **VNC ⌘-chords do not register.** `vncdo key super-h` (⌘H) was a no-op — Things stayed frontmost and visible. (Consistent with UI1's `super-a`-types-'a' finding.) So keyboard shortcuts (⇧⌘R, ⌘H, ⌘M) can't be relied on; the driver **must click menus** — which again requires Things frontmost.
4. **`open -a Things3` (the driver's necessary preamble) forces Things frontmost.** From Finder frontmost, `open -a Things3` → `lsappinfo front` = Things. If Things is hidden (⌘H) or minimized its window is off-screen and coordinate clicks can't reach its content, so the driver must re-activate it — which forces it frontmost + visible.

**Every path to driving Things ends with Things frontmost and its window visible.** Add the **LOCK1** constraint: VNC coordinate input hits the **lock screen** if the session is locked (LOCK1-f, [headless-research.md](headless-research.md)) — the ui vector requires an **unlocked** session (a VNC-reachable unlocked Mac can also be unlocked/driven by anyone with the password).

**Conclusion for the productization decision:**
- **Dedicated headless Mac** (a "closet Mac mini" nobody looks at, session kept unlocked): the focus steal is **invisible / zero-cost** — nothing is competing for the foreground. This is the viable home for the ui vector.
- **The user's own Mac**: every transform **yanks Things to the foreground and grabs keyboard focus mid-work** — this is the most-disruptive tier, and disqualifying for interactive use. A ui vector on the user's own machine would be usable only in explicit "I'm stepping away, drive it now" windows.

So the hybrid doctrine is confirmed by the mechanics: keep create/edit on the quiet vectors (tier 0, no focus steal), and gate the loud transforms (repeat manage, convert) behind a dedicated-Mac deployment.
