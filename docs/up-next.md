### ~~Approved design 2026-07-13: undo actor-scoping~~ — DONE 2026-07-14 (PR mg/undo-actor-scoping)

Shipped exactly as ratified. `selectUndoTargets` now consults the recorded actor and supports exact-token selection instead of always picking the newest N globally. The three layers:

1. **`--by <actor>` selection filter** on `things undo` + a `by` param on the MCP undo tool. Exact match (or `*` for all); `--by mcp` never matches an `undo:mcp` record (inverse mutations are their own actor and are never undo targets). The existing `--actor`/`actor` flag (who the inverse is RECORDED as) is unchanged; the --help/description now contrast the two loudly.
2. **Asymmetric defaults**: MCP undo defaults to `by: "mcp"` (an agent must pass `by: "*"` to touch the owner's edits); CLI `things undo` stays GLOBAL, with `--by mcp` to clean up after an agent. Engine + unit + MCP + CLI tests cover the "newer human record skipped under mcp scoping" case.
3. **Undo tokens**: every mutation `ok` result (JSON envelope + MCP response, including compound summaries and bounce reorders) now carries an additive `undoToken`; `things undo --txn <token>` (MCP `txn`) inverts exactly that record. **Id mechanism chosen: a compound op's shared `txn.id`, or — for a single op — a content-addressed `m-<sha256(ts+op+actor+host+uuid)[..12]>`, derived (never stored) so the write path and the trail scan always agree** (`undoToken()` in `src/audit/schema.ts`). `--txn` is mutually exclusive with `--last`/`--by` (usage error, exit 2); an unknown or already-undone token fails loud and specific (the additive `undoOf` back-reference on inverse records distinguishes already-undone from nonexistent). Also fixed in passing: the bounce-reorder summary audit record now honors `options.actor` (previously always the config actor). Doctrine: [docs/design/architecture.md](design/architecture.md) (§ Undo selection).

Optional fourth layer, still parked: session-qualified actors (`mcp:<session>`) with prefix matching on `by:` — only if parallel agents become real.

## 6. New feature items (added by Mike, 2026-07-22 triage)

Two new design-then-build items. Neither is specced yet — each wants an ideation/design round with Mike before code.

- **Reorder/sort ergonomics + optimization (containers AND the Today/Anytime views).** The shipped surface is capable but agent-hostile in shape: `things reorder <uuids…> --scope <today|evening|project|area|inbox|headings|someday|projects>` demands the caller supply an explicit uuid permutation and know the right scope/container flags, and the per-scope wire protocols (anchor-stack two-call, bounce legs, ≤10/call caps) mean a naive "resend the whole list" costs many round-trips. Mike wants (a) **optimization** of the sorting operations themselves and (b) **friendlier ergonomics** on both the CLI and the MCP API for re-sorting within a container (project/heading/area) and within the Today and Anytime views. Design directions to bring to the ideation round, not yet ratified: relative single-item placement verbs (`--before <ref>/--after <ref>/--first/--last` — `things area reorder` already speaks this dialect, so extend the dialect rather than invent a second one); accept titles/refs not just uuids (reuse the existing ref-resolution layer); a `sort --by <key>` convenience (deadline/when/title/creation) that computes the permutation server-side; and a **minimal-move planner** — diff current order vs target and emit the fewest wire moves instead of resending full lists (this is the "optimize" half; it also shrinks undo records and stays inside the per-call caps). Open questions for the round: Anytime-view ordering semantics (Anytime is a grouped/derived view — what does "reorder within Anytime" even bind to per group? needs a semantics map first, possibly a small lab probe); whether MCP gets a dedicated `sort`/`place` tool or params on the existing reorder tool; interaction with the experimental-gated native strategy vs bounce defaults.
- **Change-watch / poll mode for agents (`things watch`, name TBD).** Motivating workflow (Mike's, verbatim in spirit): Claude Code hands Mike tasks — creates a little project, schedules it into Today — then kicks off a long-running `things` CLI invocation that **watches that project specifically** and reacts when Mike actually completes/adds/removes tasks in it, without firing on unrelated library changes. Shape: a read-only, long-running CLI command (and possibly an MCP story later) scoped to one container or view (`--project <ref>`, maybe `--today`), emitting structured events — likely NDJSON lines `{event: added|removed|completed|canceled|modified, uuid, title, …}` — plus a blocking `--once`/`--until <condition>` mode that exits on the first matching event (the natural fit for an agent's "wait for change" step and for `run_in_background` Bash tasks). Implementation raw material already banked: the SYNC1 freshness proxies — `main.sqlite-wal` mtime advances on every write, so wal-mtime is the cheap wake signal; on wake, diff a scoped member snapshot (uuid set + status/stopDate/index per member) against the last one so unrelated edits are filtered out (a pure poll-interval fallback also works where the mtime signal is unavailable). Constraints to honor in design: READS ONLY (never a write path); on the prod host, reads must keep the ONE stable command/binary shape (macOS consent re-triggers on new shapes — the watcher must ride the existing library read path, not invent a new binary); define behavior when Things isn't running (DB frozen — the doctor sync-health signals apply); bounded exit conditions (timeout, `--until-complete` when every member is closed) so agent tasks can't hang forever. Open questions: event granularity for `modified` (title/when/notes edits — all or opt-in?), whether completion-of-the-whole-project is a first-class terminal event, and how undo-driven reversals should read in the event stream.
- **Container-scoped sandbox mode (added 2026-07-22).** Hard-limit an entire process's Things access to one container — the motivating case is an AREA ("this coding agent may touch only the *Work* area"), but design it as a generic **"within this container"** scope so a single PROJECT works too. Semantics Mike sketched: **reads** — every view auto-applies the filter (`things today` shows only Today items in/under the scope container; same for `upcoming`, `anytime`, `show`, search, etc. — items outside the scope are invisible, not just de-emphasized); **writes** — create/complete/edit/move allowed only for items directly or transitively inside the container (projects in the area, headings/to-dos in those projects), and any operation that targets, or would RESULT IN, an item outside the scope is refused fail-closed (moving an item out of the area = blocked; creating in the Inbox or another area = blocked or explicitly out of scope). **Trust model is the crux — this is a process-level policy, not a per-call flag**: Mike's explicit framing is "I'm not going to trust the AI to include a `--area` filter at all times." Configuration lands where the disruption gates already live — at daemon/process spawn: a flag on `things mcp` (imposed for the life of that server process, like the existing two-key write gates) and/or an env var read at startup. Design cautions to bring to the round: enforcement must live in the LIBRARY layer (one choke point both CLI and MCP inherit), not be sprinkled per-command; the boundary is only as strong as who controls the environment — the MCP-daemon-flag form is a real boundary (the agent talks only through the tools), while a bare env var on CLI-in-Bash is advisory unless the harness pins the environment (document the difference honestly; possibly refuse to silently weaken — e.g. the scoped MCP server never exposes an unscope path); ref-resolution must not leak out-of-scope matches (title search, uuid lookups, and error copy all need to answer "not found" identically for hidden vs nonexistent, or the scope becomes an oracle); undo must only invert in-scope records (actor-scoping via `--by` already exists — scope compounds with it); interactions to spec: Inbox (outside every area — invisible? a curated allow?), area-less projects, the Logbook/Trash views, `heading.convert-to-project` and other identity-replacement ops (the successor must inherit scope membership), and the §6 watch mode (a scoped process naturally yields a scoped watcher). Aligns with the fail-closed doctrine and the existing `profile`/`maxDisruption` policy-key family in `src/config.ts` — likely a sibling policy key plus spawn-time override.
