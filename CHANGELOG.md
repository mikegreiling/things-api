# Changelog

## Unreleased

- **`things todo show` and `things project show` now show inherited tags with the container each came from.** The Things app's tag filter treats an item as carrying the tags of its project and area, not just its own; the detail card already listed those inherited tags, and now it names their source. A to-do card reads `inherited: #important ‹project Renovation› #home ‹area Home›` — each inherited tag followed by a dim chip naming the project or area it is assigned to (a heading is never a source: headings cannot be tagged). A project card shows the same form for the tags it inherits from its area. The line is dim throughout — inherited tags are context, distinct from the item's OWN tags, which stay green — and it is HIDDEN entirely when there is nothing inherited (no empty placeholder, matching how the card already hides an absent `when:`/`deadline:`). Areas never show an inherited line at all (they are top-level — there is no ancestor to inherit from); an area card shows only its own tags. Inherited chips render in the app's canonical tag order (`TMTag."index"`, uuid tiebreak), the same order direct tags use, across the project+area union. Over `--json`/MCP, the `inheritedTags` array on a detail read now carries `{ tag, source: { type, uuid, title } }` per entry (previously a bare tag ref) and is ALWAYS present (an empty array when none) so machine consumers get a stable shape — additive, never merged into the direct `tags` array, which continues to mirror DB truth exactly (a tag edit still reads and restores only direct tags).
- **Lab: the two tag-inheritance axes that were modeled by analogy are now GUI-verified (developer-facing).** A disposable-VM probe (TAGINH2, screenshot oracle, no product code changed) confirmed against Things 3.22.11 that the app's tag filter behaves exactly as the read model's filter SQL assumes — in the flat-list context that filter serves. Descendant expansion is one-directional (filtering by a parent tag matches a child-tag-only item, in both a built-in list and an area scope; the reverse does not), and a heading-nested child with no direct tag filters in under both its project's and its area's tags and is excluded by "No Tag". Every modeled axis matched — no SQL change. The probe also surfaced a Things inconsistency now recorded for the report: a project's OWN in-place filter bar ignores this inheritance (an inherited-tag filter there returns nothing and its "No Tag" includes the child) — the opposite of every flat list. The library models the flat-list semantics deliberately and does not model the in-project filter bar. Details: [docs/lab/taglab-probes.md](docs/lab/taglab-probes.md) (TAGINH2), oddities §9a.
- **You can now move an area to a new position in the sidebar.** `things area reorder-sidebar <ref>` moves an area immediately above or below another area (`--before <area>` / `--after <area>`) or to the ends of the list (`--first` / `--last`); over MCP the tool is `move_area_in_sidebar`. Sidebar area order exists on no scriptable surface at all, so this drives the Things app on the machine running things-api: it is gated behind `things config set ui-enabled true` plus `--dangerously-drive-gui`, briefly moves the pointer with Things brought to the front, and needs an unlocked session with the display awake. It handles sidebars of any length — when the destination is off-screen the app's sidebar is scrolled, and very long moves are made in several verified steps; after every step the database is checked to confirm the order progressed as aimed and that the area count and every item's area assignment are untouched. If a step cannot be verified the move stops and reports exactly where the area ended up (its projects and to-dos are never affected); if the app's interface stops exposing a row's name — which can happen after heavy in-session dragging until Things is relaunched — the move refuses up front and tells you to relaunch. Undoable: `things undo --txn <token>` drags the area back to its exact previous slot. Areas whose titles are shared by another area must be renamed first (rows are addressed by their visible name). Verified end-to-end in a VM against Things 3.22.11. Details: [docs/lab/axdrag2-reorder-certification.md](docs/lab/axdrag2-reorder-certification.md).
- **Tag reordering was evaluated and deliberately NOT shipped.** The Tags window exposes tag rows without names, and tags that have never been dragged all share the same rank, so there is no reliable way to know which row is which — and a slightly mis-aimed drop would silently nest one tag under another instead of reordering it. Rather than ship a move that can corrupt a tag hierarchy, tag reordering stays unavailable; the blocking evidence is recorded for a future attempt.
- **You can now turn an existing PROJECT into a repeating one, and create a repeating project in one step.** Two new operations: `things project make-repeating <ref> --frequency <daily|weekly|monthly|yearly> --interval <1-99>` promotes an existing project to a repeating series, and `things project create-repeating <title> [--area <ref>] [--notes …] [--deadline …] [--todo …] --frequency … --interval …` creates a project and promotes it in the same call (MCP tools `make_project_repeating` and `create_repeating_project`). Like the other GUI-driven repeat controls they drive the Things app on the machine running things-api and are gated behind `things config set ui-enabled true` plus `--dangerously-drive-gui` — but unlike the project reschedule/pause/resume ops, make-repeating is **pure Accessibility** (it selects the project as a list row and opens Items ▸ Repeat…, with no pointer movement or focus steal, and works even backgrounded, where the Repeat editor detaches into its own window). Making a project repeat **replaces it** with a new repeating template (its area is kept; its schedule is normalized to Someday) and **cannot be undone**; the new template's uuid is returned. A project that has no area and is scheduled for Anytime is moved to Someday first — a cleanup-free intermediate step shown in `--dry-run` (the app normalizes the schedule to Someday during the conversion anyway, so nothing is left behind). `create-repeating` is two operations: the project is created first and **persists even if the make-repeating step refuses** (e.g. a same-titled project already occupies the row it would need to select). It refuses fail-closed on an already-repeating, trashed, or completed project, and on a title collision that would make the target row ambiguous. Recipe wired from the pure-AX row-selection findings (UIC4) and shipped **uncertified** pending an in-VM certification pass; `things capabilities` / `things doctor` show its status. Details: [docs/lab/uic4-project-selection.md](docs/lab/uic4-project-selection.md).
- **You can now reschedule, pause, and resume a repeating PROJECT's recurrence.** Three new operations join the existing to-do repeat controls: `things project reschedule-repeat <ref> --frequency <daily|weekly|monthly|yearly> --interval <1-99>`, `things project pause-repeat <ref>`, and `things project resume-repeat <ref>` (MCP tools `reschedule_project_repeat` and `set_project_repeat_state`). Like the to-do versions they drive the Things app on the machine running things-api and are gated behind `things config set ui-enabled true` plus `--dangerously-drive-gui`; unlike them, they briefly move the pointer and bring Things to the foreground to open the project's repeat menu, so they need an unlocked session with the display awake. Reschedule keeps the project's identity but cannot be undone (only a frequency and interval are supported, so an arbitrary prior rule cannot be restored); pause and resume are inverses of each other and undo cleanly. Verified end-to-end in a VM against Things 3.22.11. Details: [docs/lab/uic3-build-certification.md](docs/lab/uic3-build-certification.md).
- **Removed `todo.stop-repeat` (`things todo stop-repeat`, and the `stop` choice of the MCP `set_repeat_state` tool).** It never worked: the control that stops a to-do from repeating lives only on the opened to-do card, reachable only by a mouse double-click, which the app exposes on no scriptable surface. Stopping a *project* from repeating is likewise not offered — doing so and then reopening the demoted project crashes Things. To stop something repeating, use the app.
- **Sidebar area reorder is now a hardened, arbitrary-position recipe, and tag reorder is proven feasible (probe campaign, not yet shipped).** Building on the NATIVE1 spike, a disposable-VM campaign (AXDRAG1) worked out how to drag a sidebar area to any specific position — up or down, to the very top or bottom — and verified every move against the database's order column. Key practical findings for the eventual build: the sidebar has 40px per row-slot, and because picking up the dragged row collapses its slot, a downward drop must aim one slot higher than where the target sits at rest (upward drops aim straight at the target). Off-screen targets are reached either by synthesizing scroll-wheel input (the sidebar's scroll position is now readable, so the distance can be computed) or by Things' own edge auto-scroll during a drag, which was confirmed to work in both directions. Dropping an area onto another row can never accidentally nest it (areas are always top-level), and pressing Escape mid-drag cleanly cancels a move. Reordering is reversible by capturing the full order beforehand and dragging back. The same mechanics extend to reordering tags in the Tags window — with the caveat that dropping a tag on another tag's center re-parents (nests) it rather than reordering, so the reorder must target the boundary between rows. Nothing shipped — this hardens feasibility for a later `area.reorder-sidebar` / `tag.reorder` build. Details: [docs/lab/axdrag1-reorder.md](docs/lab/axdrag1-reorder.md).

- **Spike: the two GUI-only actions Things exposed no automation for — opening a to-do's card and reordering sidebar areas — are now proven drivable.** A feasibility spike (NATIVE1) showed that `osascript -l JavaScript` alone, through the macOS accessibility and event APIs, can resolve an on-screen element's exact position and synthesize a mouse double-click or drag at it — no separate compiled helper needed. A synthesized double-click opens a to-do's card (making its repeat controls reachable, which removes the technical blocker on `stop-repeat`), and a synthesized drag between two sidebar rows reorders an area (writing the area's order column, confirmed as `TMArea."index"`). These require the app to be frontmost and the session unlocked, and rely only on the existing Accessibility permission. Nothing shipped yet — this establishes feasibility for a later build. Details: [docs/lab/native1-spike.md](docs/lab/native1-spike.md).

- **Five of the seven ui-vector ops are now LAB-CERTIFIED; two are recorded as certification failures.** The ui vector's GUI recipes were run end-to-end through the real pipeline inside a disposable VM with Accessibility granted, against **Things 3.22.11** (`things doctor` / `things capabilities` now show a `lab-certified` tier distinct from on-device `certified`). **Lab-certified**: pause-repeat, resume-repeat, make-repeating, reschedule-repeat, and to-do convert-to-project — each observed to produce the exact database change it claims. **Still uncertified (failed)**: `stop-repeat` and `heading convert-to-project` — both need to select a specific list row, and Things exposes no accessibility or URL way to do that (a to-do's repeat card opens only on a real double-click; a heading is not selectable by the reveal URL), so the accessibility-only driver refuses cleanly and changes nothing. Certifying the five required recipe fixes that also make the shipped vector correct on real hardware: the confirmation/repeat dialogs are addressed on the app's main window (a hidden utility window otherwise sits at index 1), the repeat-frequency dropdown is now chosen by opening it and clicking the entry (setting its value silently did nothing), the interval field is found inside the dialog's content group, and the two confirmation buttons are addressed by a stable, language-independent identifier. A locale probe confirmed the vector fails closed under a non-English app language and works again once English is restored — pin the Things app to English on the deployment machine. Details: [docs/lab/uic1-certification.md](docs/lab/uic1-certification.md).

- **Compact mode: year-bearing far deadlines collapse to the bare year** (`⚑ Oct 2020` → `⚑ 2020`) — superseding their initial keep-full treatment; four digits are unambiguous against the `M/D` form. The compact floor is unchanged (the `⚑ 14d left` relative still governs the worst case).
- **Deadline flags shrink to the Things iOS narrow forms when a terminal is width-pinched.** On a narrow interactive terminal, list-row deadline tokens now switch to the compact shapes the Things iOS app uses when space is tight (Mike's screenshot oracle, 2026-07-14): a month-day absolute renders `8/12` (M/D, no zero-padding — `5/4`, `5/10`) instead of `Aug 12`, and a day-relative renders `58d ago` / `41d left` instead of `58 days ago` / `41 days left`. The `⚑` flag glyph is unchanged, and `⚑ today` stays as-is. The switch is decided ONCE per view from the effective width, never per row, so a view's right-pinned deadline gutter never mixes `Aug 12` and `8/12`. The single width floor became TWO derived floors sharing the same worst-case furniture: the existing full-form floor (`FULL_FIT_FLOOR`, 78 — byte-identical to the old `MIN_FIT_WIDTH`) and a lower compact-form floor (`COMPACT_FIT_FLOOR`, 74, budgeting the narrow deadline worst case). At width ≥ the full floor, deadlines render full (today's behavior, byte-for-byte); between the two floors, all deadlines render compact and rows fit through the normal sacrifice order; below the compact floor, the row clamps to the compact floor and the terminal wraps (the existing sub-floor rule, now at the lower floor). Year-bearing far dates (`Feb 2027`, `Mar 2025`) deliberately KEEP their full form even in compact mode — `2/27` would be ambiguous with an M/D date and the iOS oracle doesn't cover them. Fitting still engages only on a TTY: piped output, `grep`, `--json`, and the MCP server are byte-identical to before (`THINGS_WIDTH=0` disables fitting entirely; `‹date›` chips, logged dates, and detail cards are out of scope — deadline token only). Doctrine and the two-floor model: [docs/design/width-aware-tty.md](docs/design/width-aware-tty.md).
- **New: the ui write vector drives the local Things GUI to perform seven transforms that exist on no other surface — shipped UNCERTIFIED.** Making an item repeat, rescheduling/pausing/resuming/stopping a repeat rule, and converting a to-do or a heading into a project cannot be done through the URL scheme, AppleScript, or Shortcuts; this vector reaches them by driving the app's real interface through macOS Accessibility. Because it drives the real GUI, it is off by default and behind two independent switches — enable it once with `things config set ui-enabled true`, then acknowledge each call with `--dangerously-drive-gui` (`dangerously_drive_gui` over MCP). It is intended for a dedicated, always-on Mac. Its disruption is milder than screen-driving would be — on current evidence the Accessibility presses do not steal window focus and work even on a locked machine — but it still drives the live app, so a successful call notes that the change was made by driving Things through Accessibility. Setup — including granting Accessibility to the driving process and `things doctor --probe-accessibility` — is in [docs/setup.md](docs/setup.md). The seven operations: `things todo pause-repeat <ref>` and `things todo resume-repeat <ref>` (reversible — pause and resume undo each other; they keep the item and its rule, just halting or restarting new occurrences); `things todo stop-repeat <ref>` (irreversible — un-repeats the item back to a plain to-do, replacing it with a new one and leaving the already-created occurrence behind); `things todo convert-to-project <ref>` and `things heading convert-to-project <ref>` (irreversible — the original becomes a new project, a converted heading moving into its project's area with its to-dos following); `things todo make-repeating <ref> --frequency <daily|weekly|monthly|yearly> --interval <1-99>` and `things todo reschedule-repeat <ref> --frequency … --interval …` (irreversible). The five irreversible ones return the new item's id. Recurrence supports a deliberately minimal vocabulary in this first release — a frequency of daily/weekly/monthly/yearly and an interval of 1–99; weekday pickers, end-dates, and repeat-dialog reminders are not yet settable. Every one of the seven ships **uncertified**: it runs and confirms its own result, but its result carries a note that its GUI recipe has not yet been confirmed end-to-end on this build; `things capabilities` and `things doctor` show each op's certification status. MCP exposes four tools — `make_repeating`, `reschedule_repeat`, `set_repeat_state` (`pause`/`resume`/`stop`), and `convert_to_project` — each taking `dangerously_drive_gui` and `dry_run`.
- **`things --help` is now a one-screen signpost, with the detail behind `things --help` per command and `things help <topic>`.** The top-level help was a multi-scroll wall — every command's full paragraph and a ten-bullet AGENT NOTES epilog inlined into the index. It is now a grouped one-line-per-command index (Views · Browse & search · Write · Setup & diagnostics) that fits a 70-line viewport without scrolling (≤49 output lines at 100 columns), a condensed global-options block, and a footer pointing onward. Nothing was deleted — it moved: each command's full behavior still shows in `things <command> --help`, and the evicted orientation detail is now five guides reachable with `things help <topic>` — `agent` (the former AGENT NOTES), `filters` (tag/untagged/limit/since/until + the bounds doctrine), `ids` (uuids, prefixes, share links, `things <ref>` direct addressing), `output` (`--json`/width/`THINGS_WIDTH`), and `writes` (the gating model, undo, config keys, setup). `things help` with no topic shows the index; `things help <command…>` still defers to that command's own `--help`; an unknown topic lists the valid ones. The banned-vocabulary contract now also covers the topics and the index.
- **`things areas <id>` and `things projects <id>` now show that one area/project** instead of erroring with commander's cryptic "too many arguments". Given an id (or unique name, or uuid prefix) the plural list views delegate to the identical `things area show <id>` / `things project show <id>` code path — byte-identical output, a true synonym — while the bare `things areas` / `things projects` still list. The relevant show flags (`--show-later`/`--show-logged`/`--area-limit`/`--project-limit`) are accepted on the plural form and apply only when showing one.
- **Clearer excess-argument errors.** Passing a stray argument to a command that takes none (or too many) now names the command and its usage line and, for a single stray reference, suggests `things show <ref>` — e.g. `things tags X` reports `` error: `things tags` takes no positional arguments, but got: "X" `` followed by `usage: things tags [options]`, instead of the bare "too many arguments for 'tags'". Commander's did-you-mean suggestion for an unknown subcommand stays on.
- **Tags now render in the app's canonical order everywhere, not alphabetically.** A row's tags used to be sorted by title (an alphabetical guess); they now appear in the exact order the Things app shows them — ascending `TMTag."index"`, the user-draggable rank from the app's Tags window. The acceptance oracle `Replace CPAP mask & air filter` now reads `#recurring #home #housekeeping` (matching the GUI) instead of the alphabetical `#home #housekeeping #recurring`. This propagates to every surface that lists an item's tags: list rows, detail cards, area/project header tag lines, and the MCP tag inventory. **`--json` tag arrays are affected**: a tag array's ELEMENT ORDER is now canonical rather than alphabetical (the set of tags is unchanged; only the order is). Equal indexes fall back to a **UUID** tiebreak — the app's real tie-break, oracled in a VM (TAGORD1, 2026-07-15): never-dragged tags ubiquitously share rank 0, and the Things GUI orders those ties by the tag's UUID (ascending), NOT alphabetically. Proven across the Tags window, a to-do's pill row, and the filter-bar chips; `ORDER BY "index", uuid` reproduced the whole Tags-window order byte-for-byte where a title tiebreak diverged. So output stays deterministic AND matches the GUI even for the common all-tied-at-0 case. A nice consequence on a narrow TTY: because the width fitter folds tags from the end, the canonically-FIRST (GUI-priority) tags are the ones that survive the fold. The `things tags` listing keeps rendering the tag TREE depth-first (a child always follows its parent), with siblings and roots ordered by the same canonical index+uuid. Nested-tag caveat (documented, deliberately unsolved): child tags' indexes interleave globally with top-level ones — TAGORD1 confirmed `TMTag."index"` is a single GLOBAL space, not per-parent — so a flat-index sort of a multi-tag pill could in principle place a child before its parent; no live item pairs a nested tag with another, so there is no GUI oracle for that case; flat ascending index is the ratified comparator, isolated in one place (`fetchTagsForTasks`). Ruling by Mike, 2026-07-14; tiebreak corrected by TAGORD1, 2026-07-15.
- **Lint warnings now fail CI (zero-warning baseline).** All 75 outstanding oxlint warnings were fixed (54) or suppressed with a per-site reason (21), and `.oxlintrc.json` now sets `denyWarnings` (any warning is a non-zero exit) and `reportUnusedDisableDirectives: "error"` (a stale suppression is itself a failure). No runtime behavior changed.
- **`things today` and `things anytime` now keep checked-off items in place until they are swept to the Logbook, exactly like the Things app.** Completing or canceling an item does not remove it from Today/Anytime the instant you check it — the app keeps the checked row sitting in its list until its periodic "Move completed items to Logbook" sweep passes, and these two views now mirror that (the project and area cards already did). A checked item holds the exact position it had while open (its Today drag order / sidebar index is unchanged — nothing re-sorts), renders checked with the same dim / dim-and-struck styling the mixed views use, and drops out only once the log-move boundary advances past when it was completed. The Today sidebar badge is unaffected: it still counts only OPEN work, so a checked-but-unswept row shows in the list without moving the number. A project you check off likewise stays in Anytime as a single checked row that stands in for its children (they are not separately listed), matching how a logged or someday project already behaves. `--json` gains nothing new — the rows simply appear carrying their real `status` (`completed`/`canceled`) and `logged: false` — and pagination counts them like any other row. GUI-parity ruling by Mike, 2026-07-14 ("this is the behavior of the Things GUI and I like it"); the membership predicate is `OPEN_OR_UNSWEPT` in [src/read/predicates.ts](src/read/predicates.ts), the completion-≠-logged boundary model is in [src/read/log-boundary.ts](src/read/log-boundary.ts).
- **`things project show` flags a completed project that still holds open to-dos.** A project can be completed or canceled (and swept into the Logbook) while still containing OPEN child to-dos — the Things app buries those children in every live view (Today, Anytime, Inbox, Upcoming), so they surface nowhere but by drilling into the project itself. The project card now renders a muted advisory when this happens — `contains 2 unfinished to-dos — invisible in the app's live views` — so the stranded work is discoverable from the CLI. It fires only for a resolved (completed/canceled, including logged) project that holds at least one untrashed open child, and it is silent for an open project or one whose only remaining children are completed or trashed. `--json` carries the count additively as `openChildrenWhileResolved` on the project view (`0` when the advisory does not apply). Human/TTY output only; the advisory takes no reveal command because the child rows already render in the same card. Completing or canceling a project is unchanged — it still proceeds silently when the only unresolved children are in the Trash, exactly like the app.
- **`doctor` now reports a `── Sync health ──` section** — freshness and sync-liveness proxies for long-running headless operation (a "closet Mac mini"): whether the Things app is running (no app = a frozen database; there is no background sync daemon), write-ahead-log write freshness (with a staleness flag once the app is running and has not written in 24h), the last local content change (`MAX(TMTask.userModificationDate)`), the last time the app was frontmost (group-container plist `importantInformationLastForegroundDate`), and — only when a Things Cloud account is attached — the **last sync attempt** (worded as an attempt, not a confirmed round-trip: the signal advances even while offline). The cloud signal reads the account-specific `BSSyncronyMetadata` key when present and otherwise falls back to the sync-metadata timestamp nearest to now (excluding the ~now+31yr lease sentinel), so it survives the key being account-specific. With no account attached it reports "no Things Cloud account attached — without one there is no server-side sync signal to report". `--json`/the MCP `doctor` tool carry a structured `syncHealth` object (additive). All reads are read-only and go through the existing doctor database access path.
- **`things undo` learned who and which: `--by <actor>`, an asymmetric MCP default, and `--txn` exact undo.** Undo no longer always reverses the newest N changes blindly — the confusing case where an agent's undo reverted the human's just-made change is closed. Three additions. (1) **`--by <actor>`** (MCP `by`) filters which changes to undo by their recorded author — an exact name (`mike`, `mcp`) or `*` for all. It matches exactly, so `--by mcp` never touches an `undo:mcp` record (an inverse change is its own author and is never itself an undo target). This SELECTS what to undo and is distinct from the existing `--actor`/`actor`, which names who the undo itself is recorded as; both --help and the tool description now spell out the difference. (2) **Asymmetric defaults**: the MCP undo tool now defaults to `by: "mcp"`, so an agent only unwinds its own changes unless it explicitly passes `by: "*"` (or a specific author); the CLI `things undo` stays GLOBAL (the owner's Cmd+Z), with `--by mcp` available to clean up after an agent. (3) **Undo tokens**: every successful mutation result (the `--json` envelope and every MCP write response, including compound operations and bounce reorders) now carries an additive `undoToken`, and `things undo --txn <token>` (MCP `txn`) reverses exactly that one change — immune to any changes made in between. The token is a compound operation's transaction id, or for a single change a stable id derived from the change's own recorded fields. `--txn` cannot be combined with `--last`/`--by` (usage error, exit 2); an unknown token, or one whose change was already undone, fails with a specific message rather than silently doing the wrong thing. Purely additive: no envelope field was removed or changed. Doctrine in [docs/design/architecture.md](docs/design/architecture.md) (§ Undo selection).
- **Lab: the hybrid "ui" write-vector probes (UI2) + o-suite someday-reorder coverage (developer-facing).** Mapped the dedicated-Mac GUI write vector's transform set via VNC synthetic input in a disposable VM — no product code changed. Findings ([docs/lab/ui-vector-research.md](docs/lab/ui-vector-research.md)): making an existing item repeat is an IDENTITY REPLACEMENT (old uuid dies, new template + spawned instance); editing a rule PRESERVES identity (rule bytes mutate in place); Pause sets `rt1_instanceCreationPaused` (and there is NO "Stop" command — only Reschedule/Pause↔Resume/Show Latest, or an Ends bound, or delete); Convert to Project works on both a to-do AND a heading (identity replacement, heading promoted into the parent's area). Disruption profile (UI2-e): GUI driving always ends with Things frontmost + visible (menu bar belongs to the frontmost app, clicks raise Things, VNC ⌘-chords don't register, `open -a` yanks it forward) — zero-cost on a dedicated headless Mac, most-disruptive on the user's own Mac. Two DB folds: closed rows always carry a `stopDate` (UI2-g); heading-targeted placement into a completed project auto-reopens it (UI2-h). Added `lab/scripts/research-ui2.sh` and o-suite rows **O15**/**O16** locking the someday reorder two-call anchor-stack protocol (verified green in a fresh clone). Oddities §8g.
- **One disclosure-hint grammar across every list view — full echoed commands, two indentation classes.** Every muted line that discloses more content than is shown now follows a single form built by one helper (`disclosureHint` in `render.ts`): `… <count> <noun-phrase> — [label:] \`command\` [· …]`. Three inconsistencies are fixed. (1) **No more compressed ellipses** — `…71 later items` becomes `… 71 later items` (ellipsis always followed by one space). (2) **No more bare parenthesized flags or `(… — visible with …)` parentheticals** — the hidden-section hints on `area show`, `project show`, the `projects` sidebar, `--area`-scoped `projects`, and the `someday` active-projects hint now all echo the user's FULL invocation with their flags preserved (the same `hintBase` mechanism the truncation footers already used), e.g. `… 71 later items — \`things show hobbies --show-later\`` instead of `…71 later items (--show-later)`. (3) **A real noun phrase and a semantic label rule** — every count carries a noun (`later items`, `logged items`, never a bare `logged`), and a label prefixes a command only where it adds meaning the command text doesn't carry: `recent:` before an area's `--show-logged` (it reveals the recent 15, not all), nothing before an `--all`-style reveal or a logbook drill. The two hint classes are now distinguished purely by indentation: a **truncation footer** (its section is partially shown above) indents two spaces; a **hidden-section placeholder** (an entire section is unrendered) stands flush where that section would sit. The whole-view `── N more items — … ──` rail footers are unchanged (a deliberately distinct third class). `project show` now threads a `hintBase` so its hints echo the real invocation. Human/TTY output only; `--json` is unchanged. Doctrine canonized in [docs/design/render-language.md](docs/design/render-language.md) (§ Disclosure hints).
- **Fixed: the ‹waiting›/‹paused›/‹ended› chip now renders in every view, not just `upcoming`.** The repeat-status chip for a template with no materialized next date was passed explicitly by the upcoming renderer only, so the same template rendered chip-less in `area show --show-later`'s Upcoming section (and any other view). The chip is now derived inside the shared row formatter — a repeating template chips its next ‹date› when the app has materialized one, else its scheduling word — so every list view inherits it.
- **`area show --show-logged`: the truncated Logged section moved its drill hint into a footer.** The header keeps a shown-of-total count for orientation (`── Logged (15 of 251) ──`; a complete section keeps its plain count), and the actionable hint rides a muted section footer like every other truncated section — now with the READY-TO-PASTE area ref (`… 236 more — \`things logbook --area 'Hobbies'\``, previously an un-fillable `see things logbook --area` inlined into the header). The collapsed-toggle hint gained the same real ref.
- **Footer/hint copy cleanup: unlabeled `--all` escalations, and the truncated-evening hint points at `--evening`.** Two refinements to the muted footer/hint copy. (1) **The `--all` escalation drops its label everywhere.** A command like `things inbox --all` reads its own effect, so the `all:`/`full horizon:`/`everything:` label in front of it added nothing — every footer now shows the bare command (`── 950 more items — see more: \`things inbox --limit 100\` · \`things inbox --all\` ──`). This covers the flat truncation footer, the grouped per-block bottom line, and all three `upcoming` window footers. Semantic labels that DO carry information are unchanged: `see more:`, `wider:`, and the `(through <date> — …)` window phrasing. (2) **The truncated This Evening hint is now a pure section pointer.** When the default cap hides evening rows, the hint used to repeat the same quantity levers the whole-view footer directly below it already offers; it now simply points at the isolated section (`… 20 evening items — \`things today --evening\``). In `--evening` mode (where that pointer would be redundant) the hint instead keeps the `--limit`/`--all` levers. Human/TTY output only; `--json` is unchanged.
- **Fixed: fitted tag folding is now lazy, and deadlines right-align in a gutter.** Two refinements to the width-aware TTY row fitting. (1) **Lazy tag folding.** The title:tags 4:1 ratio was implemented as a hard cap, so the instant a row needed to fold it jumped straight to `#first #…` even when `#first #second #…` would have fit — a row losing a single column would collapse `#home #recurring #housekeeping` all the way to `#home #…`. Tags now render at the WIDEST progressive level that fits their available budget, dropping one tag at a time; the 4:1 ratio arbitrates ONLY under contention (when BOTH the title and the tags exceed their shares of the collapsible space). When the title is within its 4/5 share it keeps its natural width and the tags take everything left; slack that a discrete tag level does not use flows back to the title, so neither side is shrunk below what the width actually forces. The sacrifice order, container-drop timing, `MIN_FIT_WIDTH` floor, and sub-floor wrap behavior are unchanged. (2) **Right-aligned deadline gutter (experimental).** When fitting is active, a row's `⚑` deadline token is now pushed flush to the effective width, so every deadline in a view lines up in one gutter — like the GUI's right-pinned flag column. Rows without a deadline stay ragged; there is always at least one space before the gutter; and when fitting is off (non-TTY, `--json`, `THINGS_WIDTH=0`) there is NO gutter and output is byte-identical to before — the gutter lives only in fitted TTY output. It is isolated to a single function so the experiment reverts in one place. Doctrine: [docs/design/width-aware-tty.md](docs/design/width-aware-tty.md).
- **List rows fit the terminal width like the GUI — truncate, never wrap.** On an interactive terminal, a row longer than the window no longer hard-wraps (misaligning the id column); instead the title truncates with a trailing `…` and the tags fold, exactly as the Things app does, while the always-present metadata stays put. The parts that carry meaning never shrink or drop — the id column, the checkbox, the `‹date›`/`‹waiting›` chips, the `◷`/`≡`/`≔` markers and the project count chip, and the FULL `⚑` deadline token. Only the title and the tags give up space, in a ratified order (GUI-measured): title and tags share the collapsible room ~4:1, tags folding from the end into a dim `#…` marker (`#first #second #…` → `#first #…`) while the title shortens; then the inline `(container)` context — a CLI-only extra the GUI renders on a separate line — drops whole; then the last tag folds to a bare `#…`; the title bottoms out at 16 columns. There is one derived minimum width (worst-case row furniture + a 16-column title); below it the terminal wraps rather than clip anything away (the deadline is never lost). Fitting engages ONLY on a TTY: piped output, `grep`, `--json`, and the MCP server are byte-identical to before (unfitted by construction). `THINGS_WIDTH` overrides the detected width (`THINGS_WIDTH=0` disables fitting entirely). Doctrine and the collapse oracle: [docs/design/width-aware-tty.md](docs/design/width-aware-tty.md).
- **`--untagged` filters every tag-aware view to items with NO tag — the GUI's "No Tag".** It matches an item only when NEITHER a direct tag NOR an inherited one (through the project/area/heading chain) applies — the exact inversion of `--tag`: an item is untagged iff no possible `--tag X` could ever match it, so the predicate negates the WHOLE direct+inherited membership relation, not merely the row's own direct assignments (a to-do in a tagged project, or in a tagged area, or under a heading whose project's area is tagged, is NOT untagged). It is a content scope (it lifts no volume/range default) and is available everywhere `--tag` is: `today`, `inbox`, `anytime`, `someday`, `upcoming`, `logbook`, and `search`. Passing it together with `--tag`/`--exact-tag` is a loud usage error (exit 2). The MCP `read_view` and `search` tools gained a matching optional `untagged` boolean with the same exclusivity.
- **`things today` now separates This Evening from Today with a blank line.** Every other grouped renderer spaces its sections with a blank line; the `── ⏾ This Evening ──` header was pushed flush against the Today rows above it. A blank line now precedes the Evening header whenever the Today section rendered above it — and only then: in `--evening` mode the Evening header is still the first line (no leading blank). Human/TTY output only; `--json` is unchanged.
- **The list views got a canonical render design language: bold-white project titles, contextual resolved styling, chip-form repeat states.** One principle now governs how every row is styled on a terminal — a row marks only its DEVIATIONS from what its container already says, and each visual channel carries one meaning (bracket shape + bold weight = type, color = the checkbox accent and resolved marks, glyph interior = state, dim = secondary metadata, underline = heading role). Nine specific changes fall out of it: (1) **project titles render bold and default-colored (white) in every list row and every state** — never blue — routed through one exported law (`projectTitleAccent`) so the color is a one-line change; (2) a repeating PROJECT template's circle is now blue `(↻)` (it is still a project) and (3) a repeating TO-DO template's box is now plain `[↻]` (the GUI's repeat pseudo-checkbox is white); (4) a someday PROJECT's circle is now dim `(~)` (muted like a someday to-do — its type is carried by the round bracket and bold title); (5) someday project titles get the same bold white as any project; (6) **the Logbook treats resolved as normal** — completed rows render with a plain title (no dim) and canceled rows keep their strikethrough but lose the dim, while the blue `[✓]`/`[×]`/`(✓)`/`(×)` marks and the logged date are unchanged; (7) mixed views where resolved items sit among open ones (`project`/`area show`, `search --logged`/`--trashed`) keep the existing dim (completed) and dim+strike (canceled); (8) `area show`'s top projects are plain bold rows now — the underline is dropped there because they don't head a to-do group in that view (anytime keeps bold+underline, where projects genuinely head their groups; detail-card title lines keep their header styling); (9) a repeating template's `waiting`/`paused`/`ended` state renders as a dim chevron chip `‹waiting›` matching the `‹date›` chip form (the to-do detail card's prose repeat state is unchanged). None of this touches plain-text output — piped output, `NO_COLOR`, and `--json` are byte-identical; color/weight/dim are TTY-only enhancements on top of the color-strippable shape. The `things legend` table reflects the new states (including the chevron chip's widened meaning). Doctrine and the per-view roles table are in [docs/design/render-language.md](docs/design/render-language.md).
- **`things upcoming` now forecasts deadline-carrying unscheduled items, like the GUI.** A someday/anytime to-do or someday project with a future deadline and no when-date used to be invisible to `upcoming` (it only listed rows with a scheduled start date plus repeating occurrences); it now appears under its DEADLINE date, exactly where the Things app's Upcoming screen puts it. Inbox items are excluded — a future deadline does not forecast an Inbox item into Upcoming (though a DUE one still pulls it into Today) — and a dismissed-deadline item stays hidden while a re-armed (rescheduled-to-a-later-deadline) one reappears, mirroring `today`'s suppression rule one step earlier. A row that carries BOTH a when-date and a deadline still appears once, under its when-date, with the deadline as a `⚑` flag (never double-listed). Forecast rows honestly keep a null `startDate` in `--json` (no faked when-date); the merged stream groups and sorts on the earlier-appearing of the two dates, and `--since`/`--until` clip a forecast row by its deadline just as they clip a scheduled row by its start date. Rule ratified by the UPC1 lab campaign (GUI-verified); see [docs/lab/upcoming-research.md](docs/lab/upcoming-research.md).
- **`things inbox` gained `--since`/`--until`, keyed on the CREATION date.** Range bounds filter each view's own natural timeline (logbook = logged date, upcoming = scheduled appearance, changes = modification); the Inbox's is ARRIVAL, proxied by the item's creation timestamp — so `things inbox --since 2w` shows captures created in the last two weeks and `--until 2026-06` those created through June (whole-period vocabulary, `--until` inclusive through the end of the named period, exactly as `logbook`). The comparison is against the raw epoch `creationDate` column (like `changes`/`logbook`), not the packed startDate encoding. Two caveats are deliberate and accepted: a re-inboxed item (demoted from a project/area) keeps its ORIGINAL creation date, so this is arrival-into-Things rather than strictly into the Inbox; and "untouched since" (modification-keyed stale-backlog filtering) is intentionally NOT overloaded onto `--since` (that stays `things changes --since` territory). The lift rule applies as on `upcoming`/`logbook`: an explicit `--since`/`--until` drops the default `--limit 50` (an explicit `--limit` composes; `--all` conflicts with an explicit `--limit` as before). Presentation order is unchanged (the manual `ORDER BY index`), so the date filter is an invisible axis in the rows — a bound-active human run appends a dim footer note naming the effective window (`(created since Jun 29)`), which never rides `--json`. The MCP `read_view` tool is unchanged (it exposes no range params on any view). Doctrine in [docs/design/cli-grammar.md](docs/design/cli-grammar.md).
- **`things today --evening` filters the view to just This Evening.** A section visibility toggle that shows only the This Evening block (with its `⏾` header) and suppresses the Today section entirely — no Today header, no `(empty)` placeholder for the filtered-out part. The `--json` envelope keeps the today-view shape (`today` legitimately empty, `evening` populated) and its badge counts only the evening members, exactly as `--tag` narrows the view. The default `--limit 50` still applies (a visibility toggle does not lift volume defaults). New sugar: `things evening` and `things show evening` both normalize to `things today --evening` (the `≡` echo and `meta.resolvedCommand` show the canonical form). `things open evening` errors — the app has no This Evening screen behind its URL scheme — with copy pointing at `things today --evening`. The MCP `read_view` tool gained a matching `evening` boolean (today only).
- **On `upcoming` and `logbook`, an explicit `--limit`, `--since`, or `--until` now disables the OTHER's default — stating one bound hands you control of output sizing.** Previously `things upcoming` capped at both the default one-month window AND the default 50 rows at once, so a run that truncated at 50 suggested a bigger `--limit` even when the window was the real bound. Now: `things upcoming --limit 100` drops the default window (the next 100 scheduled items, full horizon); `things upcoming --until 2m` (or `--since`) drops the default 50-row cap (every item in the window); pass both to compose them. `things logbook --since`/`--until` likewise drops the default row cap. Content scopes (`--tag`) and `things changes` (whose `--since` is required) do not lift any default — their behavior is unchanged. The truncation hints follow suit: bare `upcoming` truncating inside the default window shows one line naming the window and both levers (`── 6 more items through Aug 13 — see more: \`things upcoming --limit 100\` · \`things upcoming --all\` ──`); a bare run whose row cap is not biting offers the window lever instead (`(through Aug 13 — wider: \`things upcoming --until 2m\` · \`things upcoming --all\`)`); and once you name a window the informational "(through …)" line drops, leaving only the standard row hint if an explicit `--limit` truncated.

- **CLI list rendering**: the uuid column pads to the longest uuid in the list (Things uuids vary 21–22 chars), token spacing after the `-`/`P` marker is uniform (previously the gap depended on whether dates/tags were present), and metadata is colorized when stdout is a TTY (deadline red, date yellow, tags cyan, uuid/context dim, area headers bold; `NO_COLOR`/`FORCE_COLOR` honored). Piped output contains no escape codes.


### New write capabilities (Phase 21b)

Grounded in the Phase 21b lab probe campaign ([docs/lab/phase21b-research.md](docs/lab/phase21b-research.md)).

- **Project tags** — `things project tags <uuid> --set/--add` and MCP `set_tags` now accept a project (dispatches on the item type). Full-replacement semantics mirror to-do tags; tags must name existing tags.
- **Project reminders** — `things project update --reminder HH:mm` / `--clear-reminder` and MCP `update_project`'s `reminder`/`clear_reminder`. Projects carry the same reminder codec and scope rules as to-dos (requires `--when today|evening|YYYY-MM-DD`; an existing reminder is auto-preserved on re-schedule; dated reminders are sticky).
- **Clear a tag's keyboard shortcut** — `things tag update --clear-shortcut` and MCP `update_tag`'s `clear_shortcut` (the property-delete form; `undo` now restores a cleared shortcut symmetrically).
- **Inbox reordering** — `things reorder --scope inbox` and MCP `reorder` scope `inbox` order unscheduled Inbox to-dos (native/experimental, same gate as today/project/area).

### Environment awareness & failure attribution (Phase 21a)

- **Failure attribution**: mutation failures carry an advisory `likelyCause` + hint when the signals point somewhere — AppleEvent `-1743` → `permission-denied`; a transport hang/deadline kill → `permission-pending` (the shape of an unanswered macOS consent dialog); a URL silent no-op without an authorization token → `feature-disabled` (Enable Things URLs); a Things version change since the last verified write → `app-updated`; drift blocks → `schema-drift`; residual silent no-ops → `app-behavior-change`. Surfaced in CLI stderr, `--json` envelopes (`error.likelyCause`, hint as `remediation`), and MCP tool errors.
- **Environment tuple tracking**: the Things version, macOS version, things-api version, and resolved node binary are recorded at `~/.local/state/things-api/environment.json` after every verified mutation. A successful write after a change carries a warning (a changed tuple is the classic consent re-prompt trigger), and `things doctor` reports the tuple, the last-recorded one, and the diff.
- **`things doctor --probe-automation`** (MCP: `doctor {probe_automation}`): opt-in active Automation-consent check — granted / denied (`-1743`) / pending (unanswered dialog) / app-not-running (never launches the app). Opt-in because the probe itself can summon the consent prompt; that is its onboarding use.
- **docs/setup.md**: new "Hardening against consent prompts" ladder for headless setups — the responsible-process model, FDA-to-host, the sshd identity trick, signed-binary roadmap, MDM/PPPC, and the SIP-off CI pattern.

### MCP v2 (Phase 20)

- The tool surface grows from 16 to 31 grouped semantic tools: `set_todo_status` / `set_project_status` (status + children policy + restore-children fold), promoted `move_todo` (project/area/heading/inbox/detach) and `move_project` (area/detach), `set_tags` (replace/add), `edit_checklist` (granular add/remove/check/uncheck/rename/move + stateful replace), type-generic `delete_item`/`restore_item`/`duplicate_item`, and full area/tag CRUD (`update_tag` incl. un-nest, `delete_tag` with subtree confirmation). `run_operation` + `capabilities` remain the escape hatch for the long tail.
- Server `instructions` now carry a live inventory read at server start — areas, tag hierarchy (`parent > child`), open project titles (capped at 100) — plus the reference and scheduling vocabulary, degrading to conventions-only when the database is unreadable.
- Tools carry MCP annotations (`readOnlyHint` on reads/capabilities/doctor, `destructiveHint` on permanent deletes and the generic mutation entries).
- New surface-copy contract ([docs/design/surface-copy.md](docs/design/surface-copy.md)): tool descriptions state consumer-visible behavior and side effects only — pipeline/audit/lab vocabulary is banned (regression-tested) and lives in `docs/` and the `capabilities` output. Shared parameter vocabulary moved to `src/surface-copy.ts` so CLI help and MCP schemas cannot drift apart.
- The same contract now governs the CLI and the library JSDoc: every `--help` string, the AGENT NOTES block, and the exported `ThingsClient`/operation-params docs were rewritten in consumer voice (hazard ids, probe-evidence ids, vector/tier framing, and audit/verification mechanics replaced with the behavior they imply); a help-wide banned-vocabulary test locks it in. The lab evidence remains in `docs/lab/`, the write-layer internals, and the `capabilities` output.

## 0.3.0 — 2026-07-07

The operation catalog grows from 25 to 28 kinds; every new capability is grounded in the 30-probe P-suite campaign ([docs/lab/p-suite-results.md](docs/lab/p-suite-results.md)).

### MCP server (Phase 17)

- `things mcp` serves the Model Context Protocol over stdio — 16 tools mirroring the library surface (read views incl. occurrence horizon, search, changes, item/project detail, verified mutations via dedicated tools + generic `run_operation`, batch, reorder, undo, capabilities, doctor). Hazard blocks return as structured tool errors carrying the guards' remediation text. Exposed from the library as `createThingsMcpServer()`. The SDK loads lazily — every other CLI command boots with a minimal dependency set.
- Seam hygiene: machine contracts (JSON envelope, exit codes) moved from `src/cli/` into core (`contracts.ts`); `diagnose()` and `capabilitiesTable()` promoted to library functions (the CLI `doctor`/`capabilities` commands are now thin renderers); `saveConfigKey` and the batch/undo/reorder/view option types exported from the package index.

### Project lifecycle (Phases 18–19)

- `things project cancel --children require-resolved|auto-cancel` — the URL write cascades natively (open children → canceled; completed children untouched, P01); the policy is mandatory and the cascade is verified per child, mirroring `project complete`.
- `things project reopen [--restore-children]` — reopens completed AND canceled projects (P02/P05). The bare op reopens only the project row (exactly the app's behavior — cascade-resolved children stay resolved); `--restore-children` adds verified per-child reopen legs, detecting cascade-resolved children by the <2s stopDate window (P03). Children resolved before the project are never touched (P04).
- `things project restore` — un-trashes a project IN PLACE (P06): schedule, area link, and children all keep their state.
- `undo` now reverses project complete/cancel (audit-exact child restore), project delete, and project restore.

### Container detach (Phases 18–19)

- `things todo move <uuid> --detach` — one write clears project/area/heading while the schedule is pinned unchanged (URL empty `list-id=`, P21/P22).
- `things project move <uuid> --detach` — clears the area (empty `area-id=`, P24); `project move` also gains a URL vector for regular moves (P23). Detach moves are undo-invertible from the captured prior container.

### Granular checklists (Phases 18–19)

- `things todo checklist` gains `--check/--uncheck/--add [--at N]/--remove/--rename/--to`: reads the current items+states from the DB, applies the edit in memory, and writes back through `things:///json` with per-item completed states (P18) — the only surface that recreates items pre-checked. State-preserving edits skip the H-CHECKLIST-REPLACE acknowledgement (nothing is destroyed). Checklist-item uuids are not stable across rewrites; key on title/position.
- Empty-set clears validated: `todo tags --set ""` and an empty checklist replacement (P14/P15).

### Tags (Phases 18–19)

- `things tag update --unnest` — un-nests a tag to the root of the hierarchy via AppleScript's property-delete form (`delete parent tag of tag X`, P29 — the one spelling that works; `set … to missing value` and `""` error, json `null` silently no-ops). Undo re-nests/un-nests symmetrically.
- **New hazard `H-TAG-SUBTREE-DELETE`**: deleting a tag that has child tags silently cascade-deletes the whole subtree, permanently (P16) — `tag delete` now blocks unless `--acknowledge-subtree` accompanies `--dangerously-permanent`.

### Closed permanently (documented, guarded)

Container/area removal via AppleScript `missing value`/`""` or json `null` (the URL empty-param is the sole surface); to-do↔project conversion; sidebar ordering (areas, top-level projects — reads stay available via the provisional `"index"` sort); deleting an area orphans its projects to no-area while trashing only its to-dos (P20). The app-oddities catalog (§2f, §5f, §5g) documents the report-worthy inconsistencies for Cultured Code.
