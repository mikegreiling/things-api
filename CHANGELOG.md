# Changelog

## Unreleased

- **One grammar for read routing, with a normalized-form echo.** The bare-noun shorthand, the `show`/`open` view-keyword rewrites, and the loose reference routers are now one resolver (`src/cli/resolve-invocation.ts`) applying a single documented precedence chain — registered command/alias → view keyword → reference resolution — instead of the three independent special cases (argv preprocessing, a hidden `--via-shorthand` marker, per-command keyword sets) they had accreted into. Behavior is unchanged except for three deliberate additions: (1) `things show projects|areas|tags` now works — `show` accepts EVERY list-view name, while `open` still accepts only the seven app URL-scheme ids, so `things open projects` errors with `the app has no projects list to open — open a specific project: \`things open <ref>\``; (2) a routed read reached via a sugar form (bare noun, keyword-in-show, uuid/share-link routing) prints one dim line on a TTY naming the canonical command it normalized to (`≡ things area show "Website redesign"`) — never piped, never in `--json`, and never for an already-canonical invocation; (3) `--json` envelopes of those routed reads carry the same string as `meta.resolvedCommand` (additive; MCP reads route directly and do not carry it). Also fixed: a flag-led invocation no longer skips routing — `things --json Hobbies` and `things --db <path> Hobbies` now route exactly like `things Hobbies --json`/`… --db <path>` (previously any leading flag silently bypassed the shorthand and commander errored). Only the global read flags (`--json`, `--db <path>`) are skipped over; an unknown leading flag, a flags-only invocation, and flags before a real command all keep their previous behavior. The grammar, precedence chain, keyword tables, non-goals, and the specced-but-unbuilt Phase-2 scoped verbs are written up in [docs/design/cli-grammar.md](docs/design/cli-grammar.md).
- **`things legend` explains the list views' symbols and colors.** A new read command renders the CLI's visual language grouped into sections — To-dos (`[ ]`/`[✓]`/`[×]`/`[~]`/`[↻]`), Projects (`( )`/`(✓)`/`(×)`/`(~)`/`(↻)` and the `‹remaining/total›` progress chip), Markers & chips (`★` Today, `⏾` This Evening, `⚑` deadline, `≡` notes, `◷` reminder, `≔` checklist, `⬡` area, the `‹Jul 31›` date chip, `#tag`), Colors & styles (the dim uuid prefix, dim/strike/blue/bold-underline conventions), and Sections & hints (the `──` dividers and the never-silent truncation/`--show-later` notices) — each row showing the glyph exactly as it renders, in color on a TTY. `--json` emits the same content as a `{glyph, meaning, group}` table. The legend is generated from the single glyph source (`src/cli/glyphs.ts`), so it cannot drift from what the renderers emit. `things --help` and every list view's help now point at it (`symbols & colors: run \`things legend\``).
- **The main views print a title preamble on a TTY.** `today`, `inbox`, `anytime`, `someday`, `upcoming`, `logbook`, and `trash` now lead their output with a bold view name and its Things deep link — `Anytime (things:///show?id=anytime)` — followed by a blank line, so an interactive terminal shows where you are and offers a one-click jump into the app. The preamble is strictly interactive: it is suppressed when stdout is not a TTY (so `things inbox | grep …` stays clean) and never appears in `--json`.
- **Lab harness gained an Apple Shortcuts arm (developer-facing).** The probe-suite DSL grew a `shortcut` step — the guest writes a JSON input file and runs `shortcuts run <proxy> --input-path/--output-path`, capturing the proxy's output for assertions — plus a `group:interactive` marker that the runner skips (for the delete-class proxies that re-prompt on every run and can't run unattended). `s-suite.json` is now an autonomous recurring suite in `lab:regress`, giving the two Shortcuts-only shipped operations (`heading.create`, `todo.clear-dated-reminder`) recurring live validation against the real app, alongside the already-present reorder-scope coverage in the guest e2e smoke. No product behavior changes.
- **Fixed: `things upcoming` no longer invents deadlines for deadline-less repeating to-dos.** A fixed repeat created with the repeat editor's "Add deadlines" box OFF (the GUI default) spawns instances with a start date but no deadline — yet its recurrence rule is byte-identical to a deadlined same-day rule, so the projection was reading a deadline off the rule and showing a phantom `⚑` on every occurrence of every deadline-less repeat. The deadline now gates on the template's own `deadline` column (surfaced as a new `repeating.deadlined` flag: NULL = deadline-less, a `4001-01-01` sentinel = deadlined) — the actual, VM-verified discriminator, which the rule alone cannot provide. Deadlined repeats (start-N-days-early rules, birthday-style same-day rules) still project their deadlines exactly as before. Discovered and verified by driving the repeat editor end-to-end in the VM lab (probe UI1); also corrects the analogous case for after-completion repeats.
- **The word `show` may be omitted: `things <ref>` shows the referenced item.** Any first argument that is not a registered command name resolves exactly like `things show <ref>` — `things Hobbies`, `things "Website redesign"`, `things 26cK9igr`, and pasted `things:///show?id=…` links all open the right card, with flags passing through untouched (`things Hobbies --show-later --json`). Command names are RESERVED and always win. `things show` also accepts the app's view keywords — `things show anytime` IS `things anytime` (identical flags and output), and a keyword beats a same-named project or area, mirroring the URL scheme; the typed forms (`things area show Anytime`) remain the escape hatch. Unique PROJECT names now resolve in `things show`/`things open` alongside area names (an area wins a name tie). An unknown bare word errors as `no command or item named "foo"`, covering both readings. `things open` accepts the same keywords — `things open anytime` launches `things:///show?id=anytime` directly (the URL scheme accepts these ids natively), a keyword winning over a same-named item (open those by uuid).
- **`area show` (and `things show` on an area) caps its two open-ended sections.** The project-ROWS section and the direct-to-dos section each show at most **30** rows by default — `--project-limit <n>` / `--area-limit <n>` adjust them independently, `--all` lifts both, and each truncated section trails its own exact-count muted footer echoing your invocation (`… 5 more to-dos — \`things area show Busy --area-limit 60\``). There is deliberately **no** strict total `--limit` on the detail views (their sections are containers; passing `--limit` to `area show`/`show` is a usage error naming the two knobs). The card preamble and the toggled later/logged sections are never capped. `--json` slices the same two lists and carries the grouped per-section counts additively (`meta.grouped.blocks`, kind `projects` = the project-rows section); the MCP `get_area` tool gained matching `area_limit`/`project_limit`/`all` parameters with the counts in its second result block. `project show` stays fully unbounded — headings are true containers — with its existing later/logged toggles untouched.
- **List views cap their output by default, and never drop rows silently.** The flat/chronological views — `inbox`, `today`, `upcoming`, `logbook`, `trash`, `search`, and `changes` — now show at most **50 items** by default. `--limit <n>` sets a different cap and `--all` lifts it entirely (the two are mutually exclusive; a non-positive or non-integer `--limit` is a loud usage error). When rows are hidden, a muted footer names the exact remainder and the two ways forward, echoing the flags you actually passed: `── 950 more items — see more: \`things inbox --limit 100\` · all: \`things inbox --all\` ──`. This unifies the previous ad-hoc truncation notices (trash's fetch-one-past note, upcoming's `--limit` note) into one consistent mechanism. There is deliberately **no** `--offset`/paging — escalate with a bigger `--limit` or `--all`. The cap counts items across the whole view in render order and truncates mid-group, keeping bucket/heading skeletons coherent (upcoming date buckets, logbook month headings, the Today/This-Evening split).
- **The grouped catalogues cap per block instead of cutting globally.** `anytime` and `someday` always render the **complete** sidebar skeleton — every area header and every project row — because a global top-N there would bias the output toward whichever area sorts first. They deliberately have **no `--limit`** (passing it is a loud usage error naming the right knobs). Instead: `--area-limit <n>` (default **30**) caps each area's direct list and the leading area-less block on both views; on `anytime`, `--project-limit <n>` (default **3**) caps each project's list. A truncated block trails a muted `… 14 more — \`things project show 'Website redesign'\`` (or `things area show …`) drill-down, and the view ends with one line that doubles exactly the caps that hit plus an `--all` command. `--all` lifts every cap; `--show-later` is unchanged.
- **`things someday` now mirrors the GUI's group anatomy.** Inside each group (the leading loose block and each area), someday **projects list first** — as plain rows with their `(~)` circle and count chip, flush with the to-dos around them, no header styling — then the direct to-dos in drag order (previously to-dos came first and project rows rendered like header blocks). The `--active-project-items` flag is renamed **`--show-active-project-items [n]`** (the `--show-later`/`--show-logged` family) and its output moves out of the sidebar groups into a trailing `── From active projects ──` section: a flat run of project-header blocks (underlined title, someday to-dos beneath), projects in sidebar order. The optional value caps each project's list (`--show-active-project-items 5`); bare shows every item. When the flag is off and such items exist, a muted bottom hint counts them and names the flag. Because these groups mix project rows and to-dos, a truncated group's hint splits the remainder by type — `… 1 more project, 3 more to-dos — \`things area show '…'\`` — omitting a type with nothing hidden (the per-block metadata carries the same split as additive `totalProjects`/`totalTodos`). `--json` carries the same projects-first order (children stay clustered per project; identify them by their `project` ref).
- **Truncation metadata rides `--json` and MCP, additively.** Flat views carry `meta.pagination { shown, total, limit, truncated }` (exact totals — the remainder is `total - shown`); the grouped views carry `meta.grouped { truncated, blocks: [{ kind, uuid, title, shown, total, limit }] }` — one entry per non-empty block, each with the cap that applied to it. The MCP read tools gained the matching parameters — `limit` + `all` on `search`/`changes_since` and the flat `read_view` views; `area_limit`/`project_limit` on the grouped views, and someday's `active_project_items` now accepts a number to cap that section per project — and results carry the same metadata in a second content block plus a one-line note an agent can read (`showing 50 of 1000 items — …`). No `apiVersion` bump; existing consumers are unaffected.
- **Fixed: an empty `--db` value no longer falls through to the default database.** `things inbox --db ""` used to read the default (production) database path as if `--db` had not been given; an empty or whitespace value is now a loud usage error (exit 2).
- **MCP `read_view` someday accepts `show_active_project_items`.** The parameter is now named to match the CLI's `--show-active-project-items` flag; the original `active_project_items` name stays accepted as a silent alias, so existing callers are unaffected.
- **Time-window flags are standardized on `--since`/`--until`.** `things upcoming` gains `--since` (skip occurrences before a date) alongside its `--until`, both taking the shared relative/absolute period vocabulary (`2w`/`3m`/`1y`, `2026-09`, `2026-09-15`, `2026`); `--all` on upcoming now means no date bound *and* no row limit. `things changes --since` accepts that same period vocabulary in addition to a full ISO date-time. The shared grammar lives in `src/surface-copy.ts` so the CLI help and MCP schemas describe it identically.

## 0.8.0 — 2026-07-12

- **`things capabilities` now shows each operation's undo classification.** Every op entry carries an `undo:` line — `reversible`, `reversible-with-loss`, `conditional`, or `irreversible` — with the note explaining what `things undo` can and cannot restore (and the acknowledgement flag its inverse needs, where one applies). The same data rides the `--json` envelope and the MCP capabilities tool as an additive `undo` field, straight from the test-locked reversibility matrix.
- **Fixed: `--when` no longer accepts the raw URL `@time` grammar silently.** `--when 2026-07-20@09:30` used to flow straight into the URL — the app would SET the date and reminder while verification asserted the literal string as the date, reporting a false mismatch on a write that succeeded. The write layer now validates `when` strictly (`today | evening | anytime | someday | YYYY-MM-DD`), and the CLI splits a well-formed `@HH:mm` suffix into when + reminder for the commands that take both (`todo add`, `todo update`, `project update`) — giving both alongside each other, or a malformed suffix, is a clean usage error.
- **`things projects` lists every sidebar area, even project-less ones** — an area with no (visible) projects renders its `── ⬡ Area ──` header with a muted `(no projects)` beneath, completing the sidebar mirror.
- **Default-hidden rows are never silent.** Every view that hides items behind a toggle now says so with a muted count naming the flag: `things projects` trails each group with `…7 later projects` (an area whose every project is later still surfaces as a header + count) and ends with `(23 later projects — visible with `things projects --show-later`)`; `project show` and `area show` add `…n later items (--show-later)` and `…n logged (--show-logged)` hints; `things trash` gained the loud `--limit` truncation note the other bounded views already had. Hints are human-render only — `--json` shapes are unchanged.
- **Fixed: children of a trashed project no longer leak into live views.** Project deletion is shallow (only the project row is flagged; children's Trash membership is derived through the container chain), and `upcoming`, `today`, and `search` checked only the row's own flag — so a trashed project's dated child still appeared in Upcoming and would have surfaced in Today when its date arrived. All live views now exclude the whole derived-trash chain (heading and effective project both untrashed; `anytime`/`someday` already did). `search --trashed` widens back to the chain. The exceptions are deliberate: `things trash` lists directly-flagged rows, and a trashed project's OWN view (`things show <uuid>`) still shows its would-be-recovered children — now with a `(trashed)` marker on the card — plus a `── Trashed ──` bucket for directly-trashed ones. That last part fixes a black hole the app itself has (filed as oddity 6½): trash a to-do AND its project separately in the GUI and the to-do becomes invisible in every view — the CLI keeps it findable in both `things trash` and the project's view.
- **`things projects` hides later projects by default, like the sidebar.** Someday and future-scheduled projects no longer appear unless you pass `--show-later`, which appends them AFTER each group's active block (loose block / area) rather than intermingling them — scheduled projects first in Upcoming order (date ascending, the app's drag order within a day), then someday in drag order — someday rows carry the `(~)` mark, scheduled rows their `‹date›` chip. A scheduled project whose date has arrived counts as active (the same membership test as Anytime). Applies to `--json` (`later: true` in the library API).
- **Every write operation now carries a declared reversibility class, locked by a per-op test matrix.** A new source-of-truth table (`src/write/reversibility.ts`) classifies all 37 operation kinds as `reversible`, `reversible-with-loss`, `conditional`, or `irreversible` — derived from and cross-checked against what `things undo` actually does. It is a TOTAL record over the op catalog, so a newly added operation fails to compile until it is classified, and an exhaustiveness guard in the new `test/unit/reversibility-matrix.test.ts` suite fails until that op also has a do/undo round-trip, an anti-clobber precondition probe, and (for conditional/irreversible ops) both branches proven. This is the systematic replacement for the scattered scenario-based undo tests that let `todo.clear-dated-reminder` ship mislabeled irreversible. A cross-check test keeps the table's `irreversible` rows and undo's internal irreversible catalog from ever drifting apart. Developer-facing only — the classification is not yet surfaced in any command (a `things capabilities` reversibility column is deferred).
- **Granular checklist edits now record INTENT, and `things undo` reverses them surgically.** A single `things todo checklist <uuid> --check/--uncheck/--add/--remove/--rename/--move-item` (and `edit_checklist` over MCP) is audited as its own operation (`todo.edit-checklist-item`) capturing the action and the targeted item's before/after state — not a snapshot of the whole list. Undo then applies a TARGETED inverse against the checklist as it stands *now* (a 3-way merge): it reverts only the item you changed and leaves every other item — including anything you checked off in the app since — exactly as it is. If the item you originally changed has itself moved out of band (someone toggled, renamed, or removed it), or a duplicate title makes it ambiguous, undo refuses that step rather than guessing. Previously, undoing any checklist change replayed a titles-only snapshot that reset every completion state and clobbered concurrent edits. WHOLESALE `--item` replacement keeps snapshot semantics but is upgraded: undo now restores per-item completion state too (via the `things:///json` form), and refuses if the list changed at all since the replacement. Honest limit unchanged: the forward edit is still read-then-rewrite, so an out-of-band edit landing during that window is overwritten by the forward write — there is no item-level surface to prevent it.
- **`things upcoming` shows the next month by default.** The full dump (often hundreds of rows) scrolled the useful near-term items off screen; now the view is bounded by `--until` (default `1m`), with a trailing `(through Aug 12 — --all for the full horizon)` note whenever a bound is active. `--until` takes relative periods — `2w`, `3m`, `1y` from today — or absolute ones (`2026-09-15`, `2026-09`, `2026`, through the period's end); `--all` restores the app's full Upcoming. `--limit <n>` caps rows after the date bound (same loud truncation note as `things logbook`). The dateless trailing `── Repeating To-Dos ──` section always survives — a date bound cannot apply to rows with no date. Applies to `--json` too. `things logbook --since` gains the same relative vocabulary (`--since 2w` = the last two weeks; whole periods like `2024-03` now bound at the period's START instead of parsing as an instant).
- **`things projects` mirrors the sidebar.** Projects now list in the app's sidebar order — loose (area-less) projects first in their drag order, then each area by its sidebar rank with its projects grouped beneath a `── ⬡ Area ──` header (the redundant `(Area)` suffix dropped) — replacing the raw global-index dump that interleaved areas. `--area <ref>` output stays a flat list. `--json` keeps the flat shape, now in sidebar order.
- **Clearer diagnosis when a macOS Automation dialog is waiting for a click.** An unanswered Automation consent prompt makes every object-targeting AppleScript command time out with AppleEvent `-1712` while URL-scheme commands keep working — which previously read as an app hang. `things doctor --probe-automation` now classifies the in-band `-1712` as consent-pending (not just our own probe timeout), and both the doctor detail and the write-failure hint now say who the dialog is addressed to — your terminal app (macOS attributes AppleScript to the terminal emulator, e.g. Ghostty, not to `things`), that it renders on the machine's physical screen (easy to miss over SSH/remote sessions), and how to recover if it was dismissed (System Settings > Privacy & Security > Automation).
- **Apple Shortcuts is now a first-class write vector**, unlocking two operations that exist on NO other surface. `things heading add <project> <title>` (also `create_heading` over MCP, op `heading.create`) creates a heading inside an existing project — impossible via the URL scheme and AppleScript. `things todo clear-reminder <uuid>` (`clear_reminder`, op `todo.clear-dated-reminder`) removes the time-of-day reminder from a date-scheduled to-do while keeping its date — the only in-place, schedule-preserving way to do this (a URL `when=today`→re-date bounce and an AppleScript move-to-Inbox also clear the reminder, but each shifts the item's schedule as a side effect; same-date dated reminders are otherwise sticky). Both are delivered through the bundled Things proxy shortcuts: run `things setup shortcuts` once first, and grant each its one-time "Always Allow" on first run (headless after that). If a required proxy is missing the command is blocked up front with a `things setup shortcuts` remediation — nothing is dispatched. Creating a heading has no automatic undo (heading deletion is interactive-only), so `things undo` reports it irreversible. These paths are seam-tested; end-to-end validation against a live library is deferred (Mike smoke-test, or a future VM sitting once the lab runner grows a Shortcuts arm).
- **`things todo clear-reminder` no longer needs the Shortcuts proxies for a non-repeating dated to-do, and is now reversible.** When the proxies aren't installed the command falls back to a pure-URL bounce (it briefly re-schedules the to-do to Today to drop the reminder, then re-dates it back — non-atomic, and if it's interrupted between the two steps the message tells you the exact residual state and how to finish). Repeating to-dos still require the proxies (a URL re-schedule crashes a repeating template), and are blocked up front with the `things setup shortcuts` remediation rather than bounced. `--vector shortcuts|url-scheme` forces a delivery path. `things undo` now reverses a cleared reminder — it re-attaches the reminder to wherever the to-do is scheduled *now* (not the date it was cleared from, so an out-of-band re-schedule is preserved); if the to-do is no longer on a date, or has become a repeating template, undo reports it irreversible instead of guessing.
- **`things undo` gained a precondition guard against clobbering out-of-band edits.** Before applying any inverse, undo confirms the fields it would overwrite still hold the value it recorded right after the original change; if something (you, or the app) has since edited one of those fields, undo refuses that step and stops rather than silently overwriting the newer value. This is on top of the existing verified read-after-write.
- **Fixed: the bundled `things-proxy-find-items` shortcut now actually searches.** The previous file echoed its input back instead of querying Things (its Find Items action shipped without a predicate). The repaired shortcut filters on Title (case-insensitive exact match, first match returned; no match returns empty output) and was validated in the VM lab before shipping. If you installed the old copy, delete it in Shortcuts and run `things setup shortcuts` again — the replacement asks for its own one-time permission on first run. No pipeline operation is affected; the proxy is diagnostic-only.
- **Recurring rows and projects read more like the app.** A repeating template now seats its `↻` glyph INSIDE the box — `[↻]` for to-dos, `(↻)` for projects — where a checkbox would be (the app shows only the refresh mark on the rule row, never a checkbox), instead of trailing the box as a separate mark; spawned instances keep ordinary boxes. Open project rows render their circle and title blue, matching the app's list accent — a color-independent cue on top of the round bracket that a row is a project.
- **List rows speak the app's visual language.** To-dos carry a state checkbox — `[ ]` open, `[✓]` completed, `[×]` canceled (title struck through on a TTY), `[~]` someday — and projects a round counterpart: `( )` untouched, quarter pie `(◔ ◑ ◕)` while in progress, `(◉)` all children done, `(✓)`/`(×)` completed/canceled, `(~)` someday. Future-scheduled rows show the app's date pill as a dim `‹Jul 31›` chip (year appended when not the current year) instead of `@2026-07-31`; logged rows show their completion date (`[✓] Jun 15 Old win`); rows with notes carry a `≡` marker; project rows show the remaining-count chip `‹12›`. The `[completed]`/`[canceled]` text markers and the `P`/`-` type markers are retired — shape carries the state, so piped/`NO_COLOR` output stays unambiguous; color, dim, and strikethrough are TTY-only enhancements.
- **Tags render green** (the app's tag color), correcting the earlier blue.
- **`things today` marks every row** — yellow `★` in Today, cyan `⏾` in This Evening — and grouped views (`anytime`) use the crescent for effective This-Evening members instead of the star.
- **`project show` and `area show` render later items inline.** Scheduled, repeating, and someday rows now appear under their heading (or at the end of the active block) with their state carried by the `[~]` box and date chips, instead of being pulled out into a separate "Later" section that disassociated them from their headings. `--hide-later` omits them, mirroring the app's toggle. JSON shapes are unchanged.
- **Flat views label heading-nested to-dos with their parent project** (the app's behavior). Read model: to-dos gain an optional computed `headingProject` ref in list views (`project` itself stays null for heading-nested rows — DB truth); additive, non-breaking.
- **Completion and logged are now separate states, like the app.** A closed item joins the Logbook only when the app's log-move sweep passes it (`TMSettings.logInterval`/`manualLogDate` — no per-row column records this; boundary model live-verified on a Daily-interval library). `things logbook` excludes closed-but-unswept items; they stay CHECKED IN PLACE in `project show`/`area show` (under their heading or in the active block), and closed-unswept projects appear in the area view. Entities gain a `logged: boolean` (additive). The v26 schema fingerprint baseline was recomputed for the two new TMSettings columns.
- **`things logbook` grew into the archive tool**: `--area` (direct items + its projects' children, heading-nested included — headings themselves never appear), `--project` (all children incl. heading-nested), `--since`/`--until` (`2024` and `2024-03` cover the whole period), GUI-style month headings (`── July ──`, year appended beyond the current year), and a loud trailing notice whenever `--limit` truncated the range.
- **Deadlines render like the app's flag**: `⚑ 3 days left` (bold gray) up to 14 days out, `⚑ 4 days ago` (bold red) up to 59 days past, `⚑ today`, then date shorthand (`Sep 16` / `Feb 2027`), positioned at the END of the row after tags and container. Closed rows and repeating templates drop the flag (template rows store a 4001-01-01 sentinel, now guarded everywhere); no-date templates whose rule assigns deadlines show a bare `⚑`.
- **Project circles simplified; progress moved to a ratio chip.** The pie glyphs are gone (they render inconsistently at cell size): circles use the shared marks — `( )` `(✓)` `(×)` `(~)` — and rows show `‹remaining/total›` (`‹0›` when childless).
- **Row indicators after the title, GUI order**: `◷` reminder, `≡` description, `≔` checklist. List-row tags render muted (the app's gray pills); tags are green only on the opened resource. The Today `★` / Evening `⏾` marks moved after the checkbox (app position), the crescent is bright blue, and areas carry a green `⬡` (headers, `area show`, and `things areas`, which now shows uniform muted uuid prefixes).
- **Detail cards restyled** (`todo show`, `project show`, `area show`): type-labeled header (`To-Do:`/`Project:`/`Area:`/`Heading:`), a `uri:` line with the resource's `things:///show?id=` share link (pastes back into any ref argument), labeled `when:`/`deadline:`/`logged:`/`area:`/`project:`/`heading:`/`tags:`/`inherited:`/`repeating:` lines with absent values hidden, the full note as a text block, and the checklist under its own heading. `when:` collapses past schedules to `★ Today`/`⏾ This Evening` (app semantics) and appends the reminder time; `status:`/`start:` lines retired (the box and `when:` carry them; `(trashed)` stays on the title). Repeating templates show `↻ <next date>` and `deadline: ⚑ set per occurrence (+n days)` instead of sentinel dates.
- **`things show <ref>` and `things open <ref>`** — loose routers: paste any uuid, ≥6-char prefix, share link, or area name; to-dos/projects/areas route to their views (a heading ref shows its containing project), tags and checklist items are rejected, ambiguous prefixes list candidates. `open` launches the Things app on the resource (explicitly NOT headless). `project open`/`area open`/`todo open` are type-checked mirrors. Library: `client.read.showTarget(ref)`.
- **Composite views default to the app's toggles**: later rows are hidden unless `--show-later` (replacing `--hide-later`; `area show` gains the GUI's date-ordered `── Upcoming ──` — scheduled projects, to-dos, and repeating templates intermixed — and `── Someday ──` with its leading projects block, with active projects now at the top of the view). Logged items appear only with `--show-logged [n]` (bare = 15 recent for areas with a pointer to `things logbook --area`; full logbook for projects), replacing the old silent 10-row cap.
- **`things upcoming` mirrors the app**: within-day order is the app's drag order (todayIndex — live-verified; also applied to `project show`/`area show` scheduled groups), GUI-style date headers with decaying granularity (`── Wed Jul 15 ──` for the next week with per-row date chips suppressed, `── Jul 19–31 ──`, month headers through next year, then bare years; empty periods absent), and the trailing `── Repeating To-Dos ──` section: templates with no set next occurrence, with their `waiting`/`paused`/`ended` status (derived from the pause flag and the decoded rule's end conditions). JSON: additive trailing rows with `startDate: null` and the decoded `repeating.rule`.

## 0.7.0 — 2026-07-09

## 0.6.0 — 2026-07-09

### Reference resolution & hidden internals (mildly breaking)

- **Things share links are accepted wherever a uuid or name is expected.** Right-click → Share → Copy Link in the app yields `things:///show?id=<uuid>`; paste it directly into any `--uuid`/`--area`/`--project`/`--tag` argument and it is stripped to the id automatically — no intermediate step.

- **Names resolve in tiers** (areas, tags, projects): exact uuid → exact title → case-insensitive title → normalized title (whitespace/dash-insensitive, NFC) → uuid prefix. The first tier with exactly one match wins definitively; several is an ambiguity error. `--area family` finds `Family` without tripping over `Family - Jennifer`; `on-hold` finds `On Hold`. **Leading emoji/symbols are significant** — a name beginning with an emoji must be typed with it (so an archived `🗄️errand` tag is never matched by a bare `errand`). See [docs/design/reference-resolution.md](docs/design/reference-resolution.md).
- **Area/tag references now accept a unique uuid prefix** too (≥6 chars), like task uuids.
- **Internal sort keys are no longer surfaced.** `index` and `todayIndex` are dropped from task/heading/area/tag responses; checklist items are now `{ title, status }` only (their uuid, index, owning-task, and timestamps were unstable implementation details — a checklist item's uuid is regenerated on every rewrite and was never addressable). Ordering is conveyed by array order; `--json` still carries full task uuids. **This changes read-response shapes.**
- **Checklist edits target by title OR 1-based index** (`--index` / MCP `index`). Duplicate titles resolve best-effort — `check` picks the first unchecked match, `uncheck` the first checked, others the first match — with `--index` as the exact override. Checklist items have `open`/`completed`/`canceled` states (canceled is read-only; the write path produces open/completed) and no trashed/logged state — they travel with their parent to-do.

- **Safety: project operations now reject wrong-type targets.** `project.update`, `project.complete`, `project.set-tags`, and `project.delete` previously accepted any task uuid — passing a to-do or heading uuid resolved cleanly and compiled a project-shaped command around the wrong row (undefined app behavior; a heading in a schedule-class specifier is known to crash Things). Every `project.*` op now verifies the target is a project and points a mis-typed uuid at the right command family, matching the existing guards on `todo.*` and `heading.*`. Cross-table uuids (an area passed to a task/project op) were already caught as not-found.

### Short uuids

- **Every uuid parameter accepts a unique PREFIX (minimum 6 characters)** — CLI, MCP, and library alike (`things todo complete Vwn2yoRP` instead of the full 22-character id). Resolution is an indexed range scan; an exact full uuid always wins (a 21-char uuid can prefix a 22-char one), unknown prefixes report not-found, and ambiguous prefixes fail with every candidate listed. `things undo` and audit records always carry full uuids.
- **List output shows shortened uuid prefixes** — the shortest length unique within the list, never below 8 characters so a copied prefix stays unique database-wide, not just list-wide. `--json` envelopes and detail-view headers carry full uuids unchanged.

### Heading operations + transactional undo

- **`things heading rename / archive / unarchive`** (ops `heading.rename`/`heading.archive`/`heading.unarchive`; MCP `rename_heading`/`archive_heading`/`unarchive_heading`) — AppleScript by-id addressing (lab P10/P10b/P11), no Shortcuts setup. Archive is the preferred way to retire a heading (row deletion exists only interactively behind a per-run consent dialog); it is reversible. With open children, `--children` is required: `complete` and `cancel` ride the app's own cascades (one atomic call each; already-resolved children are never touched); `reparent` moves children to the project root first, keeping them open. `unarchive --restore-children` reopens the children the cascade resolved (matching timestamps; a someday child comes back as someday — lab-verified).
- **Compound operations now undo as one unit.** Bounce reorders and reparent-archives group their steps under a transaction: the individual legs no longer appear as separate undo targets, and undoing the operation reverses the whole sequence (a bounce reorder inverts to a single reorder back to the recorded pre-order; a reparent-archive un-archives the heading and moves every child back under it).
- **Crash guard confirmed necessary**: the lab verified that AppleScript `schedule` on a heading row crashes Things (process death). All todo operations reject non-to-do targets.
- **`things area show <ref>`** (MCP: `get_area`): composite area view mirroring the native UI — the area's direct to-dos (active first), its projects in sidebar order, later (scheduled/repeating/someday), and logged items. Targets by uuid or unique name.
- **`--area` filters accept area names** (uuid or unique title, ambiguity errors) everywhere — `things projects --area Hobbies` now works; `--tag` already resolved titles.
- **Safety: every todo operation now rejects non-to-do targets.** Passing a HEADING uuid to a todo op previously reached the app unchecked; URL writes silently no-op on heading rows, and an AppleScript `schedule` on one is a suspected app crash (lab P10b). Projects are pointed at the project commands.
- **CLI colors**: tags render blue (matching the app's tag color), not cyan.
- **Area tag clearing is now a validated surface** (lab P10e): `things area update <ref> --tags ""` removes every tag.
- **`things reorder --scope someday` now also orders area-less SOMEDAY PROJECTS** (one kind per call — to-dos and projects never mix). Someday projects are sidebar rows, so this orders the sidebar's someday segment natively. The two row types stack in opposite directions inside the app's Someday handler (lab P8b vs P9e), so the compiler emits the matching validated two-call protocol per type.
